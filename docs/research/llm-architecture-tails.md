# LLM architecture — decision tails

## Recommendation

Adopt the Vercel AI SDK v6 (`ai@6.0.208`) as the provider-neutral LLM layer, with a `createProviderRegistry` of direct provider SDKs (`@ai-sdk/anthropic@3.0.85`, `@ai-sdk/openai@3.0.73`, `@ai-sdk/google@3.0.83`, `@ai-sdk/openai-compatible@2.0.51` for local). Keep `pricing.ts` as the synchronous cost-of-record (generalized to `"provider:model"` keys) and give it a named owner plus a CI staleness alarm. Capture per-call `{provider, model, usage, costUsd, raw_usage_json}` through ONE `wrapLanguageModel` middleware into a new `llm_call` trace table; compute `run.cost_usd` as a read-time `SUM(llm_call)` (not an incremental update) to avoid a fan-out race. Make structured output uniform with `generateText({ experimental_output: Output.object({ schema }) })` plus a mandatory Zod re-validate/repair net, authoring the six stage types once in `src/domain/stages.ts` and modeling graph edges as `{from,to}` objects rather than tuples to sidestep Gemini `prefixItems` variance. Keep the `code` stage as a free-text `complete()` and put an explicit validation contract at the `code`→`critic` boundary. A workflow_version arm is `{stage: {provider, model, params}}` stored in the existing `model_snapshots` column — the call site and schema are identical across arms, though `params` (effort/thinking/cache) is partly Anthropic-specific, so an Anthropic-vs-GPT arm differs in more than {provider, model}. Runner-up is a self-hosted LiteLLM gateway behind a near-unchanged `complete()` (pick it only if authoritative per-call USD or virtual-key budgets dominate). Treat claudeception as a dev-only `claude -p` subprocess escape hatch — never an in-pipeline transport — because Anthropic's Feb-2026 ToS forbids OAuth-subscription tokens in app/Agent-SDK requests; tag its estimate-sourced cost as a distinct `cost_source` and exclude it from eval verdicts. Do an Anthropic typed-output-with-thinking spike (vercel/ai #7220/#9351/#13355) BEFORE declaring the design done, since adaptive thinking conflicts with the tool-forcing that structured output requires.

## Migration from current code

One PR, zero existing call sites to rewrite (stages are not yet wired to `complete()`), but ~7-9 engineer-days of new code dominated by the middleware and an Anthropic spike. Sequenced, each step testable in isolation: (1) ~0.5d — add and pin `ai@6.0.208` + the four `@ai-sdk/*` packages; run the "ai v6 + providers v3" skew sanity check. (2) ~0.5d — generalize `src/llm/pricing.ts` key from `model` to `"provider:model"`, add OpenAI/Gemini/local entries, keep cache multipliers Anthropic-only, add a named owner + a CI assertion that the in-file `cached_at` is <90 days old. (3) ~2-3d (largest) — build `createProviderRegistry` + the single `wrapLanguageModel` middleware: per-provider usage normalizers with recorded `result.usage` fixture tests, store `raw_usage_json` verbatim, make a thrown/failed insert write a visible `cost_usd=NULL, error` marker row rather than dropping silently, and make the wrapped registry the only exported entry point (lint-forbid importing provider packages outside `src/llm/`). (4) ~1d — add the `llm_call` table + migration keyed by run_id+stage+workflow_ver with an idempotency key (run_id+stage+call_ordinal); compute `run.cost_usd` as a read-time `SELECT SUM(cost_usd) FROM llm_call WHERE run_id=$1` (never read-modify-write) so concurrent researcher fan-out can't race; FIRST resolve open question #3 (does eleatic read `llm_call` directly or only via `run.eleatic_run_id`) because it gates the table's ingestion shape. (5) ~1d — port `Plan`/`Research`/`PrereqGraph`/`GatedGraph`/`PageSpec`/`CritiquedArtifact` from TS interfaces to Zod in `src/domain/stages.ts`, modeling `edges` as `z.object({from,to})` (not `z.tuple`) to avoid Gemini's `prefixItems`-only-via-`responseJsonSchema` edge, adapting to the TS tuple at the boundary. (6) ~1d — implement `complete`/`completeObject` over the registry with the Zod re-validate + one bounded repair-retry net; keep `code` on `complete` and add the `code`→`critic` artifact-validation contract. (7) ~1d, high-information, do BEFORE sign-off — Anthropic typed-stage spike for the thinking-vs-tool-forcing conflict (#7220/#9351 middleware workaround) and the Zod-keyword 400 stripping (#13355); if Anthropic typed-output-with-thinking can't be made reliable, `graph`/`spec`/`critic` on Anthropic need a thinking-off or text+parse fallback. The Anthropic-only knobs (adaptive thinking, `output_config.effort`, `cache_control`) move into `StageModel.params` / `providerOptions.anthropic`.

## Claudeception plan

Dev-only escape hatch via a `claude -p` subprocess adapter, never an in-pipeline transport and never composed into the gateway. Anthropic's Feb-2026 ToS forbids routing app/Agent-SDK requests through Pro/Max OAuth (enforcement Jan 9 2026), so a subscription cannot legitimately back any provider in the pipeline; the gateway plane is API-key-only and is also out of bounds for OAuth. Implement an `anthropic-cli` provider option behind the same `StageModel` resolver that shells out to `claude -p --output-format json`, parses `total_cost_usd`, and records it — but tag every such `llm_call` row with `cost_source='claude_p_estimate'`. Two correctness rules: (1) `total_cost_usd` is a CLIENT-SIDE estimate from a bundled price table, not authoritative billing, so estimate-sourced rows must NEVER be compared against `pricing.ts`-sourced rows in an eval verdict (the same split-brain hazard flagged for gateways) — they are for local sanity only; production/CI eval runs use real API keys + `pricing.ts`. (2) `claude -p` draws from the developer's NORMAL Pro/Max subscription limits (the separate-credit-pool change was paused 2026-06-15, never enacted), so a heavy local pipeline sweep can exhaust personal limits and block the developer's own interactive Claude Code — cap dev-loop iterations or point heavy sweeps at a cheap real API key. This composes with the multi-provider design because it is just one more `StageModel.provider` value resolved by the same registry/middleware, isolated by `cost_source`, with no path into shared/CI runs. The `ANTHROPIC_BASE_URL` OAuth-proxy alternative (zero code change) is NOT recommended — it is ToS-sensitive if it forwards subscription tokens; if any shared or CI path could ever touch it, get written confirmation first.

## Decision questions

- Cost-of-record: self-computed pricing.ts (deterministic, synchronous, you maintain rates + a 90-day CI staleness alarm) vs gateway-reported USD (self-reporting per-call but async + markup)? Recommendation: self-computed now; add a gateway only when a concrete flip-on criterion fires (provider count >~5 or a missing first-party package; OR a re-run with corrected rates flips an arm ranking; OR you need per-key spend caps).
- How does eleatic ingest llm_call — a direct read of the table, or only via run.eleatic_run_id and the run-level SUM? This is load-bearing and gates the migration: a direct read needs llm_call's columns to match eleatic's per-call schema; the indirect path lets llm_call stay app-private. Resolve BEFORE building the table (migration step 4).
- Direct provider SDKs vs a gateway in prod: do you need one-key access / virtual-key budgets / 100+ models enough to take on a gateway dependency, markup, and infra to run/secure? If not, direct SDKs + registry is simpler and cheaper.
- Which stages truly need typed structured output vs free text? graph/spec/critic need typed JSON; code emits HTML and stays a free-text complete() call. Confirm the code->critic boundary contract (validate the PageArtifact shape before the typed critic, since CritiquedArtifact embeds the free-text html).
- claudeception transport for dev: claude -p subprocess (ToS-safe, separate process, estimate cost, draws on personal subscription limits) vs ANTHROPIC_BASE_URL OAuth proxy (zero code change but ToS-sensitive if it forwards tokens)? Recommendation: subprocess; get written confirmation if any shared/CI path could touch it.
- How are the non-portable Anthropic knobs (effort / adaptive thinking / cache_control) represented in a workflow_version so an Anthropic arm and a GPT arm remain comparable — dropped on non-Anthropic arms, or mapped (e.g. 'effort: high' to a provider-equivalent notion)? This is the residue of the fact that an arm differs in more than {provider, model}.
- Should structured-output repair rate be a first-class eval metric? If weak/local-model arms hit the Zod repair-retry path more often, the extra tokens/latency land in the same cost/quality ledger the eval pillar compares — recording repair_attempts/repaired per llm_call defends the apples-to-apples claim and turns repair rate into a quality signal rather than hidden noise.

## Risks

- Middleware is the single writer of the cost-of-record and a SPOF: if it throws after a paid call, mis-normalizes one provider's usage, or is bypassed by a direct generateText, an arm's cost ledger is silently wrong. Mitigated by writing a visible NULL+error marker row on insert failure, storing raw_usage_json for after-the-fact recompute with per-provider fixture tests, and making the wrapped registry the only exported entry point — residual risk is an undetected normalizer bug on a low-traffic provider, bounded by retained raw usage.
- Structured-output repair-retries contaminate the eval comparison: weaker/local-model arms that fail Output.object more often burn extra tokens + latency that land in the very cost/quality ledger the eval pillar exists to compare, so 'apples-to-apples' is not free. Mitigate by recording repair_attempts/repaired per llm_call, counting repair tokens in cost_usd, and reporting repair rate as a first-class arm metric.
- Anthropic typed output conflicts with the wrapper's adaptive thinking via TWO distinct real bugs: structured output forces tool_choice but extended thinking requires tool_choice auto|none (vercel/ai #7220, #9351 workaround), AND @ai-sdk/anthropic forwards Zod JSON-Schema keywords the API rejects with a 400 (vercel/ai #13355, which the native SDK strips). graph/spec/critic on Anthropic may need a thinking-off or text+parse fallback — spike this before sign-off.
- run.cost_usd roll-up can race under researcher fan-out: many concurrent llm_call inserts feed one run aggregate, so an incremental read-modify-write loses updates. Mitigated by computing run.cost_usd as a read-time SUM(llm_call) (or an atomic UPDATE ... = cost_usd + $delta in the same transaction) plus an idempotency key on llm_call so a retried insert can't double-count.
- Stale pricing.ts silently flips eval-arm rankings, and it has no owner or alarm today (the code comment is a passive 'update when Anthropic pricing changes'). Mitigated by a named owner, in-file source URLs + check date, and a CI assertion failing when the snapshot is >90 days old; recoverable because raw_usage_json lets historical costs be recomputed at corrected rates. Note self-hosted LiteLLM does NOT remove this — only hosted OpenRouter self-reports per-call.
- Meta-SDK lock-in: adopting the AI SDK couples the LLM layer's upgrade cadence to a fast-moving library (generateObject was deprecated within one major). Exit cost is bounded (the depended-on surface is small: generateText + Output.object + wrapLanguageModel + registry, with roll-our-own as a known fallback) but is a real coupling to weigh, not zero.
- claudeception cost is an estimate AND a split-brain hazard: claude -p total_cost_usd is a client-side estimate, not authoritative billing, so mixing it with pricing.ts-sourced USD in one ledger corrupts comparisons. Mitigated by a cost_source tag and excluding estimate rows from eval verdicts.
- Subscription-path operational lock-in for DEV: claude -p draws from the developer's normal Pro/Max limits (the separate-credit-pool change was paused 2026-06-15), so a heavy local pipeline sweep can exhaust personal limits and block the developer's own interactive Claude Code — not just a billing footnote. Mitigate by capping dev-loop iterations or pointing heavy sweeps at a cheap real API key.
- Provider-package version skew (ai v6 + @ai-sdk providers on the v3/v2 line) is correct but surprising; pin all versions and test together.
- Anthropic-only knobs (adaptive thinking, output_config.effort, cache_control) have no portable OpenAI/Gemini equivalent; they live in StageModel.params, which is why an Anthropic arm and a GPT arm differ in more than {provider, model} and need an explicit representation decision to stay comparable.
