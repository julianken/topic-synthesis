# LLM architecture — per-approach profiles (with verification)

## Vercel AI SDK  
*trust: high* — The Vercel AI SDK (`ai` package, current stable v6, released 2025-12-22) is a TS-first unified LLM layer: one `generateText`/`streamText` call swaps provider+model by passing a different model instance or string id, with Zod/Standard-Schema structured outputs and per-call token usage returned on every result — making per-stage swap and A/B arms trivial — but it returns tokens, not USD, so the eleatic cost trace keeps using your own pricing map (or the AI Gateway's generation-info lookup for true USD).

**howItWorks:** App code centers on two functions from the `ai` package: `generateText`/`streamText` (and the deprecated-but-still-working `generateObject`/`streamObject`). A "model" is a `LanguageModelV2` instance produced by a provider factory; you swap providers by swapping that instance. Provider packages (separate semver, install alongside `ai`): `@ai-sdk/anthropic` (createAnthropic), `@ai-sdk/openai` (createOpenAI), `@ai-sdk/google` (createGoogleGenerativeAI), `@ai-sdk/openai-compatible` (createOpenAICompatible — for local/vLLM/Ollama/LM Studio/any OpenAI-shaped endpoint), plus first-party `@ai-sdk/gateway` and community providers (e.g. `@openrouter/ai-sdk-provider`). Recommended shape for topic-synthesis: replace the single `complete()` Anthropic wrapper with (a) a `createProviderRegistry({...})` that registers each provider under an alias, and (b) keep `STAGE_MODELS` but widen each entry from a bare Claude model-id string to `{provider, model, providerOptions}`; a stage runner resolves that to a model instance via the registry and calls `generateText`. Each provider factory accepts `baseURL` and `headers`, so a claudeception proxy or local endpoint is a config change, not a code fork. Current major: **AI SDK v6.0** (npm `ai`, e.g. 6.0.x as of mid-2026; v5 released 2025-07-31 still widely supported and Gateway-compatible). NOTE low confidence on the exact patch version — verify with `npm view ai version`.  
**structuredOutputs:** Strong and provider-uniform. v6 unifies on `generateText({ output: Output.object({ schema }) })` / `streamText` with an `output` setting; `generateObject`/`streamObject` still exist but are deprecated in v6 (removed in a future major) — for a new/refactored layer, write the v6 `Output.object` form. Schemas: Zod is first-class; v6 also accepts any library implementing the Standard Schema / Standard JSON Schema interface (Valibot, ArkType, raw JSON Schema via `jsonSchema()`), so each stage gets a typed, validated JSON object. The SDK abstracts the provider-native mechanism for you — it uses Anthropic tool-use, OpenAI json_schema/structured-outputs, and Gemini responseSchema under the hood — so the same `Output.object({schema})` call yields the same typed result across Claude/GPT/Gemini, which is exactly what you want when a stage's schema must hold constant across A/B arms. Caveat: not every model supports strict structured output equally (some open/local models via openai-compatible are weaker); validate-and-retry still matters per stage.  
**costAndUsageCapture:** Per-call TOKEN usage is returned on every result as `result.usage` (`{ inputTokens, outputTokens, totalTokens, ... }`); v6 renamed the cache/reasoning detail fields — `cachedInputTokens`/`reasoningTokens` are deprecated in favor of `usage.inputTokenDetails.cacheReadTokens` and `usage.outputTokenDetails.reasoningTokens`. USD cost is NOT returned by the core providers — only tokens — so for the eleatic trace you keep computing USD yourself from a per-(provider,model) pricing map (your existing `pricing.ts` pattern, generalized beyond Claude). The ONE place USD is exposed natively is the AI Gateway: `result.providerMetadata.gateway.generationId` lets you call `gateway.getGenerationInfo({id})` for a real `totalCost`/`upstreamInferenceCost` (USD), and some responses surface `providerMetadata.gateway.cost`/`marketCost` — but that requires routing through Gateway and is an extra async lookup, so the self-computed pricing map remains the most reliable, synchronous source for the trace. Wire usage capture cleanly via SDK middleware (`wrapLanguageModel` + `LanguageModelV2Middleware`) so every stage call logs `{provider, model, usage, costUsd}` into eleatic without per-call-site code.  
**perStageSwapAndAB:** Excellent fit. A `workflow_version` maps each stage -> `{provider, model, providerOptions}`. Two clean mechanisms: (1) `createProviderRegistry({ anthropic, openai, google, local })` then resolve `registry.languageModel('openai:gpt-5.x')` per stage — aliases keep version configs declarative and serializable; or (2) AI Gateway string ids like `'anthropic/claude-sonnet-4.6'` / `'openai/gpt-5.x'` / `'google/gemini-...'` passed straight to `generateText({ model })`, with per-request routing via `providerOptions.gateway` (`order`, `only`, `tags`, `user` for spend attribution). Because the call site is identical and only the model instance differs, an A/B arm is purely data: swap the stage's model entry, keep the same prompt + same `Output.object` schema, run both arms, diff the typed outputs and the captured usage/cost. The model identity per stage is itself a serializable axis of the version — exactly the "which model powers each stage" lever you want recorded in eleatic.  
**claudeceptionFit:** Good fit, and it composes cleanly because auth/endpoint is per-provider config. For dev, point `@ai-sdk/anthropic` at a local proxy that fronts your Claude Code OAuth subscription: `createAnthropic({ baseURL: 'http://localhost:PORT', headers: {...} })` (the provider also honors `ANTHROPIC_BASE_URL`, and supports `authToken`/`ANTHROPIC_AUTH_TOKEN` for `Authorization: Bearer` instead of `x-api-key`). The SDK itself has NO built-in Claude-subscription/OAuth path — you must run an Anthropic-protocol-compatible (or OpenAI-compatible, via `@ai-sdk/openai-compatible`) local proxy that bridges to your subscription; the SDK just needs the baseURL/headers. Production swap is one env/config flip back to the real API key / Gateway. Important caveat: routing through a subscription proxy means the proxy, not a metered API, returns usage — so your token numbers (and thus eleatic USD) are only as accurate as what the proxy reports; treat dev-run costs as estimated, and reserve real-API or Gateway runs for cost-sensitive evals.  
**bestFitWhen:** Best fit when you want ONE TS call site across many providers, per-stage model selection is a serializable data axis of a workflow_version, structured JSON-per-stage must stay constant across A/B arms, and you're already on Next.js (same vendor). It is the default recommendation here. Lean on the optional AI Gateway when you also want native USD-per-call, provider fallback/routing, and a single key for 200+ models; skip the Gateway and keep your own pricing map + middleware if you want zero external dependency and fully synchronous cost capture. The claudeception dev path works via baseURL to a local proxy — adopt it for dev iteration, but run cost-sensitive evals against real API/Gateway for trustworthy USD.  

**Strengths:** One unified call site (generateText/streamText) across Anthropic, OpenAI, Google, Mistral, Bedrock, Groq, OpenRouter, and any OpenAI-compatible local model — swapping a stage's provider is swapping a model instance, no per-provider client code; Per-stage swap + A/B is pure data: createProviderRegistry aliases or Gateway string ids let a workflow_version serialize stage->{provider,model} and run arms with identical prompts/schemas; Provider-uniform structured outputs via Output.object + Zod/Standard-Schema — same typed-JSON contract per stage regardless of which model backs the arm (SDK maps to Anthropic tool-use / OpenAI json_schema / Gemini responseSchema); Per-call token usage returned on every result; first-class middleware (wrapLanguageModel) to log usage+computed cost into eleatic centrally; baseURL/headers/authToken per provider make the claudeception proxy and local/OpenAI-compatible endpoints config-only, composing with the multi-provider design; Optional AI Gateway gives true USD per call (getGenerationInfo), provider fallback/routing (order/only), BYOK, and 200+ models behind one key without rewriting app code; TS-first with strong types; native to Next.js (same vendor); large ecosystem, active releases

**Weaknesses:** Core providers return tokens, not USD — you still own a per-(provider,model) pricing map for the eleatic cost trace (USD is only native via the Gateway's extra getGenerationInfo lookup); v6 deprecated generateObject/streamObject; a refactor should adopt generateText+Output.object now to avoid a second migration when they're removed; Abstraction can hide provider-specific knobs your current wrapper relies on (adaptive thinking, output_config.effort, cache_control breakpoints) — these move into providerOptions.anthropic and must be re-mapped per provider; cross-provider parity for thinking/effort/caching is uneven; No built-in Claude-subscription/OAuth — claudeception needs an external proxy you run and trust; usage/cost from that proxy is only as accurate as the proxy reports; Two separately-versioned package families (ai core vs @ai-sdk/* providers) — version-skew can break provider interfaces on upgrades; pin and test; Structured-output reliability varies for weaker open/local models behind openai-compatible; per-stage validate-retry still required; Telemetry has had gaps (e.g. usage span attributes on generateText/generateObject) — don't rely solely on OTel spans for cost; capture usage from the result object in middleware

**Verification:**
- [supported] Current major is AI SDK v6.0 (stable, released 2025-12-22); v5 (released 2025-07-31) remains supported and Gateway-compatible. Exact npm patch of `ai` unverified (LOW confidence on patch). — correction: Patch now confirmable: `ai@latest` is 6.0.208 (published 2026-06-18T01:01Z); v5 latest is 5.0.204. A v7 line also exists in beta (7.0.0-beta.181) / canary, which the claim did not mention but does not contradict 'v6 is the current stable major'.
- [supported] Core providers return token USAGE only, NOT USD cost; native USD requires the AI Gateway (result.providerMetadata.gateway.generationId -> gateway.getGenerationInfo({id}).totalCost). Self-computed pricing map remains the reliable synchronous cost source.
- [supported] In v6, generateObject/streamObject are DEPRECATED in favor of generateText/streamText with Output.object({schema}); usage fields cachedInputTokens/reasoningTokens are deprecated in favor of usage.inputTokenDetails.cacheReadTokens / usage.outputTokenDetails.reasoningTokens.

## Model-router gateways (LiteLLM / OpenRouter)  
*trust: high* — Put a single multi-model gateway behind your complete() wrapper so stages swap models by string and you get usage+cost back per call — LiteLLM (self-host, OpenAI- AND Anthropic-compatible, returns x-litellm-response-cost) for control and claudeception-style key isolation, OpenRouter (hosted, one key, usage.cost inline) for zero-ops breadth; neither lets you back a route with a Claude subscription anymore.

**howItWorks:** Replace the hardcoded `@anthropic-ai/sdk` call inside complete() with one HTTP client pointed at a gateway base URL; the rest of the app is unchanged. Model selection becomes a string in the request body, so a stage's {provider,model} is just config.

LiteLLM (self-hosted proxy / "AI Gateway", BerriAI/litellm, Python proxy): you run `litellm --config proxy_config.yaml` (or Docker), define a model_list mapping aliases -> upstream provider+model+key. It exposes an OpenAI-compatible surface at `/chat/completions` (callable from the official `openai` npm SDK with baseURL set to the proxy) AND an Anthropic-compatible surface at `/v1/messages` and `/anthropic/*` pass-through (callable from `@anthropic-ai/sdk` by overriding baseURL — minimal churn to your existing wrapper). 100+ providers (OpenAI, Anthropic, Vertex/Gemini, Bedrock, Azure, vLLM/Ollama for local). Virtual keys, per-key budgets, logging, load-balancing built in. Upstream provider API keys live only in the proxy config — the app holds one virtual key. Confirmed via docs but I did not pin the exact current litellm version number — LOW confidence on a specific version string.

OpenRouter (hosted SaaS, not a package you run): one OpenRouter API key, OpenAI-compatible `/api/v1/chat/completions`, 100+ models selected by `model: "anthropic/claude-...", "openai/gpt-...", "google/gemini-..."`. Use the `openai` npm SDK with baseURL `https://openrouter.ai/api/v1`. No infra to run; OpenRouter holds the upstream relationships and bills you in credits.

Recommended shape for topic-synthesis: keep complete() as the single seam; let a workflow_version specify, per stage, a gateway target string. Run LiteLLM locally for dev/eval (key isolation + notional cost even for proxied calls) and either keep LiteLLM in prod or point at OpenRouter for breadth.  
**structuredOutputs:** Both gateways pass an OpenAI-style `response_format: { type: "json_schema", json_schema: { name, strict: true, schema } }` through to the underlying model, giving you ONE schema call site across providers — this is the main parity win of a gateway. OpenRouter explicitly documents structured outputs for OpenAI (GPT-4o+), Anthropic (Sonnet 4.5 / Opus 4.1+), all Gemini, Fireworks, and most open models, recommends `strict: true`, and supports streaming partial JSON; it warns support is per-model and suggests `require_parameters: true` so unsupported models error rather than silently ignore the schema. LiteLLM supports Structured Outputs / JSON mode via `response_format` on its OpenAI-compatible surface, and on its Anthropic `/v1/messages` surface it transforms OpenAI `response_format` into Anthropic's native output_format and injects the required `structured-outputs-2025-11-13` beta header. Caveat: a gateway only normalizes the REQUEST shape; the strictness/validity guarantee is still the underlying model's, so per-stage you should validate the returned JSON (e.g. zod) regardless. Anthropic's own typed-output beta header is recent — MEDIUM confidence that `structured-outputs-2025-11-13` is the current header string.  
**costAndUsageCapture:** This is where gateways shine for the eleatic trace — cost is returned to you, not computed by you. OpenRouter returns the authoritative per-request USD cost INLINE in every chat-completion response: `usage` carries `prompt_tokens`/`completion_tokens`/`total_tokens` plus `cost` (in credits, == USD) and `cost_details` (e.g. upstream_inference_cost, cache_discount), with `prompt_tokens_details.cached_tokens` and `completion_tokens_details.reasoning_tokens`. The old `usage:{include:true}` / `stream_options:{include_usage:true}` opt-ins are now deprecated/no-ops — usage is always included. You can also re-query by generation `id` at `GET /api/v1/generation`. LiteLLM returns per-call cost in the response HEADER `x-litellm-response-cost` (USD), and persists prompt/completion/total tokens + spend to its LiteLLM_SpendLogs table, queryable via `/spend/logs`. So both feed per-call usage+cost straight into eleatic; the only integration nuance is OpenRouter=read response body, LiteLLM=read a response header (your fetch wrapper must surface headers, which the raw `openai` SDK hides — use the SDK's raw-response accessor or a plain fetch).  
**perStageSwapAndAB:** Clean. Because the app speaks ONE API and the model is a string, a workflow_version pins per stage simply as `{ stage: { target: "litellm-alias" | "anthropic/claude-...", response_format } }`. Swapping planner from Claude to GPT to Gemini is a one-line config change with zero code churn — exactly the A/B/eval-arm axis the product wants. LiteLLM adds first-class load-balancing/routing (multiple deployments under one alias, fallbacks), which is useful for cost/latency A/B but is orthogonal to the version-level model pinning you control in app config. Keep the per-stage pin in YOUR workflow_version (so eleatic records exactly which {provider,model} produced each trace) rather than delegating choice to gateway-side routing, otherwise the eval arm becomes non-deterministic.  
**claudeceptionFit:** This is the sharp edge. As of April 4, 2026 Anthropic blocked third-party harnesses (the "OpenClaw ban") from using Claude Pro/Max SUBSCRIPTION quota via OAuth — subscription-backed routing through tools/proxies now requires switching to pay-as-you-go API billing or a real API key. So you CANNOT durably back a gateway route with the Claude Code subscription anymore. LiteLLM's "Claude Code Max Subscription" tutorial still documents OAuth forwarding (`forward_client_headers_to_llm_api: true`, client sends both `x-litellm-api-key` and an `Authorization` OAuth header) with no ToS warning, but that path is the exact pattern Anthropic now blocks — treat it as unreliable/non-compliant, LOW confidence it keeps working. Practical answer: gateways are an API-KEY plane, not a subscription plane. Keep claudeception OUTSIDE the gateway — for local dev, point Claude-targeted stages at a local Anthropic-compatible shim (`claude -p` / a local OAuth proxy) by setting that stage's baseURL, and have complete() choose baseURL by environment. Production swaps that stage's target to the gateway with a real key. The multi-provider design composes with this precisely because baseURL+model are already per-stage config; claudeception is just one more (dev-only) baseURL value.  
**bestFitWhen:** You want minimal change to the existing complete() seam, you need per-call cost handed to you for eleatic without maintaining a pricing map, and per-stage model choice is config not code. Pick self-hosted LiteLLM when you need private/local eval runs, virtual-key budgets, local models (vLLM/Ollama), and the ability to keep your @anthropic-ai/sdk wrapper via its Anthropic-compatible surface. Pick OpenRouter when you want zero ops, the widest hosted catalog, and authoritative inline USD cost with a single key/bill. Either way, keep the Claude-subscription (claudeception) dev path OUTSIDE the gateway as an env-selected baseURL, because the gateway plane is API-key-only post-OpenClaw-ban.  

**Strengths:** One API surface, model-as-string: per-stage swap and A/B arms are config-only, zero app churn — ideal for the workflow_version/eval-arm model.; Cost is RETURNED, not computed: OpenRouter usage.cost inline (USD); LiteLLM x-litellm-response-cost header + SpendLogs — direct, authoritative feed into the eleatic trace.; Uniform structured output: one response_format json_schema call site normalized across OpenAI/Anthropic/Gemini/open models.; Key isolation: app holds one gateway/virtual key; upstream provider keys live in the gateway (LiteLLM) or are abstracted entirely (OpenRouter).; LiteLLM is self-hostable and OSS (local/private eval runs, vLLM/Ollama for local models); OpenRouter is zero-ops with the widest hosted model catalog and a single bill.; LiteLLM speaks BOTH OpenAI- and Anthropic-compatible surfaces, so your existing @anthropic-ai/sdk wrapper can keep working by only changing baseURL.

**Weaknesses:** Neither can back a route with the Claude Code SUBSCRIPTION — Anthropic's Apr 4 2026 third-party-harness ban means subscription-via-gateway is dead; gateways are API-key-only. Claudeception must live outside the gateway.; Structured-output guarantee is still the underlying model's; the gateway only normalizes the request shape, so per-stage JSON validation (zod) is still mandatory and feature support is per-model (use require_parameters/strict to fail loudly).; Added hop: LiteLLM is infra you must run/secure/upgrade (a Python proxy + a DB for SpendLogs); OpenRouter adds a vendor in the data/billing path and a credit-pricing layer.; Cost-read ergonomics differ: LiteLLM cost is in a HEADER the plain openai SDK hides — your wrapper must surface raw response headers.; OpenRouter cost is in credits and can include routing/markup nuances; for an authoritative eval ledger confirm credit==USD and capture cost_details, don't recompute from a local price map.; Version/beta-header drift risk: Anthropic's structured-output beta header and gateway versions move fast (LOW/MEDIUM confidence on exact strings) — pin and re-verify.

**Verification:**
- [supported] As of April 4, 2026 Anthropic blocked third-party harnesses (the 'OpenClaw ban') from using Claude Pro/Max SUBSCRIPTION quota via OAuth; a Claude subscription cannot back a gateway route, so gateways require provider API keys / pay-as-you-go. LiteLLM's Max-subscription OAuth-forwarding tutorial predates/contradicts this and should be treated as unreliable. — correction: Minor framing nuance: the ban is specifically about third-party tools piggybacking on subscription OAuth tokens at flat-rate pricing; Claude Code / Claude.ai / Claude Desktop still work on a subscription. Gateways were never able to use the subscription via a sanctioned path; they use API keys, which is unaffected.
- [mostly-supported] OpenRouter returns per-request USD cost inline in the chat-completion response as usage.cost (plus usage.cost_details), and the older usage:{include:true} / stream_options:{include_usage:true} opt-ins are now deprecated/no-ops (usage always included); LiteLLM returns per-call cost in the x-litellm-response-cost response HEADER. — correction: OpenRouter usage.cost is denominated in OpenRouter credits (1:1 USD-pegged), not an explicitly-labeled USD field. LiteLLM's x-litellm-response-cost header is reliably present only for non-streaming calls; it is currently omitted for streaming responses (open issue).
- [supported] Both gateways accept OpenAI-style response_format:{type:'json_schema', json_schema:{strict:true,...}} and pass it through per model; LiteLLM's Anthropic /v1/messages surface transforms it to Anthropic output_format and injects the structured-outputs-2025-11-13 beta header. — correction: Confidence on the beta header string 'structured-outputs-2025-11-13' should be HIGH (confirmed by Anthropic's own docs). The feature has since reached GA (output_format relocated to output_config.format, beta header optional), but the beta header path LiteLLM injects still works during the transition window.

## Native thin provider abstraction (roll our own)  
*trust: medium* — Generalize the existing complete() wrapper into a thin LlmProvider interface over the three native SDKs (@anthropic-ai/sdk, openai, @google/genai) plus one OpenAI-compatible adapter for local/gateway/OpenRouter — lowest abstraction tax, closest to current code, you keep full control of structured-output normalization, the pricing map, and the claudeception base-URL hook.

**howItWorks:** Define one interface mirroring today's complete(): `complete(opts) => { text, usage: {inputTokens, outputTokens, cacheRead?, cacheWrite?}, costUsd, stopReason }`, plus a per-stage `{provider, model}` resolver replacing the current STAGE_MODELS map (src/llm/models.ts) and a `responseSchema` field carrying a JSON Schema. Then 3-4 adapter modules behind it:

- Anthropic adapter = today's client.ts almost verbatim. Versions in repo: @anthropic-ai/sdk 0.104.2 (latest is also 0.104.2). Keeps `thinking:{type:'adaptive'}` + `output_config:{effort}`; structured output ADDS `output_config.format` (see below) — same object, composes cleanly.
- OpenAI adapter = `openai` 6.44.0 (latest). Use `client.responses.parse(...)` (Responses API, with `zodTextFormat`) or `client.chat.completions.parse(...)` (Chat Completions, with `zodResponseFormat` from `openai/helpers/zod`). Usage on the response: `prompt_tokens` / `completion_tokens` (Chat) or `input_tokens`/`output_tokens` (Responses).
- Gemini adapter = `@google/genai` 2.8.0 (latest; this is the current unified Google SDK that replaced the deprecated `@google/generative-ai`). `ai.models.generateContent({ model, contents, config:{ responseMimeType:'application/json', responseSchema } })`; usage in `response.usageMetadata` (`promptTokenCount`/`candidatesTokenCount`).
- OpenAI-compatible adapter = reuse the `openai` client pointed at a different `baseURL` to cover Ollama, vLLM, OpenRouter, or a gateway. One adapter, N endpoints. Ollama and vLLM both expose `/v1/chat/completions`; vLLM does JSON-schema guided decoding; OpenRouter is fully OpenAI-compatible at `https://openrouter.ai/api/v1`.

A `workflow_version` row pins `{provider, model, effort, schemaName}` per stage; the eleatic trace records the resolved tuple + usage + costUsd per call. ~4 adapters, each ~40-80 LOC; you already own pricing.ts and the cost math, so marginal effort over the existing wrapper is modest.  
**structuredOutputs:** Three genuinely different native shapes you must normalize to one typed-JSON contract:
- Anthropic: `output_config.format = { type:'json_schema', schema:{...} }`. GA (no beta header) since Sonnet 4.5; supported on Opus 4.8 / Sonnet 4.6 / Haiku 4.5. NOTE the deprecated `output_format` top-level param has MOVED to `output_config.format`. Anthropic's `strict:true` is a TOOL-USE flag, not a response-format flag (do not copy OpenAI's response-format `strict` here). Because the current code already sends `output_config:{effort}`, adding `format` is a one-key extension of the same object.
- OpenAI: `response_format:{ type:'json_schema', json_schema:{ name, schema, strict:true } }`, or let `zodResponseFormat`/`zodTextFormat` + `.parse()` generate it. `strict:true` requires `additionalProperties:false` and all keys `required`.
- Gemini: `config.responseMimeType:'application/json'` + `config.responseSchema` (an OpenAPI-subset schema, NOT full JSON Schema — propertyOrdering, limited keyword support; complex/recursive schemas can fail).

Practical normalization: author each stage schema once in Zod, derive JSON Schema (zod-to-json-schema) for Anthropic/OpenAI, and down-convert to Gemini's OpenAPI subset (the riskiest edge). Validate every provider's returned JSON against the same Zod schema in the wrapper so the typed-JSON contract is enforced uniformly regardless of provider — preserving today's "downstream stages trust the wrapper" invariant.  
**costAndUsageCapture:** You already own the authoritative path: pricing.ts maps USD/MTok and estimateCostUsd() computes cost from returned usage, including Anthropic cache read (~0.1x) / write (~1.25x) multipliers. Generalizing means: (1) normalize each SDK's usage object into the existing TokenUsage shape (Anthropic input/output/cache_read/cache_creation; OpenAI prompt/completion or input/output + cached_tokens in prompt_tokens_details; Gemini promptTokenCount/candidatesTokenCount), and (2) extend MODEL_PRICING with OpenAI/Gemini/local rates keyed by `provider:model`. Local models (Ollama/vLLM) cost $0 — record usage, set costUsd 0. OpenRouter is the one provider that RETURNS cost: usage accounting is now always included in every response (the old `usage:{include:true}` opt-in is deprecated/no-op), so you can record OpenRouter's reported cost directly instead of from your map. This keeps eleatic's per-call USD trace uniform across metered, free-local, and gateway calls.  
**perStageSwapAndAB:** Clean. Replace the const STAGE_MODELS map with a workflow_version record that stores, per stage, `{provider, model, effort, maxTokens, schemaName}`. The orchestrator resolves stage -> tuple -> adapter at call time, so an A/B/eval ARM is literally a different workflow_version row; swapping the planner from Opus to GPT-5 or Gemini for one arm is a data change, not a code change. Because the wrapper validates all providers against the same Zod schema and writes the same trace shape, comparison is apples-to-apples: same prompt, same typed output contract, differing only in {provider,model}. This abstraction is YOURS, so the per-stage axis maps directly onto the eval pillar with no framework assumptions in the way — the main cost is that you, not a library, must keep each adapter's structured-output + usage mapping correct as SDKs move.  
**claudeceptionFit:** Composes well, because the swap point is per-adapter config, not global. Two supported routes for the Anthropic adapter in dev, both leaving OpenAI/Gemini/local on their own metered keys:
1. Base-URL proxy: the official Anthropic TS SDK honors `ANTHROPIC_BASE_URL` (and `baseURL` in the constructor) — point it at a local proxy backed by your Claude Code/Max subscription; flip the env var (or pass nothing) to hit api.anthropic.com with a real key in prod. Zero app-code change. CAUTION: a proxy that EXTRACTS/forwards the OAuth token violates Anthropic's consumer ToS (tightened Feb 2026; third-party harnesses blocked from Max limits Apr 4 2026) — mark this route ToS-sensitive.
2. Subprocess `claude -p` (Claude Agent SDK in print mode): the supported programmatic pattern. `claude -p --output-format json` returns a ResultMessage carrying `total_cost_usd` and token usage, and it natively supports structured output via `outputFormat` (TS) / a JSON Schema. This is the safer claudeception path: a fourth "anthropic-subprocess" adapter that shells out, parses the JSON envelope, and maps total_cost_usd straight into the trace. Note (June 15 2026): subscription `claude -p` usage draws from a separate monthly Agent SDK credit pool (Pro $20 / Max5x $100 / Max20x $200) before metered rates. Either route is just one adapter swapped behind the interface; the other providers are untouched.  
**bestFitWhen:** Best when the provider set is small and fairly stable (3-4 providers), the team values control and minimal dependencies over breadth, structured-output schemas per stage are mostly flat objects (Gemini-friendly), and you want the eval/cost trace and claudeception path to be first-class and bespoke rather than adapted from a framework's abstractions. Reconsider (lean AI SDK or a gateway) if you expect to add many providers/models quickly, need provider-agnostic streaming/tool-calling/retry out of the box, or want one library to own structured-output normalization for you.  

**Strengths:** Lowest abstraction tax and closest to current code: client.ts already IS a complete() wrapper with usage+cost, so the Anthropic adapter is near-verbatim and you only add the per-stage resolver + 3 sibling adapters.; Full control of the three things that matter most here: structured-output normalization to one Zod-validated typed-JSON contract, the USD pricing map you already own, and the trace shape eleatic consumes.; No framework lock-in or version-coupling to a meta-SDK (AI SDK / LangChain); SDK upgrades are localized to one adapter.; claudeception composes naturally — it's an Anthropic-adapter-only concern (base-URL env or claude -p subprocess), leaving OpenAI/Gemini/local on metered keys.; One OpenAI-compatible adapter (just a different baseURL on the openai client) covers Ollama + vLLM + OpenRouter + any gateway, so 'open/local models' is mostly free once OpenAI is done.; OpenRouter returns its own per-call cost, giving a ground-truth cross-check against your pricing map.

## Cross-cutting decision concerns  
*trust: high* — Adopt the Vercel AI SDK 6 as the per-stage provider abstraction, layer a thin gateway (AI Gateway or OpenRouter) for one-key access and authoritative USD cost, keep your own pricing map as the fallback cost-of-record for the eleatic trace, and treat "claudeception" as a DEV-ONLY local-CLI escape hatch (NOT an in-pipeline provider) because Anthropic's Feb-2026 policy bans routing app/Agent-SDK requests through Pro/Max OAuth.

**howItWorks:** Wrap the current hardcoded Anthropic `complete()` behind a provider-neutral interface and back it with the Vercel AI SDK (`ai` package, major v6, released 2025-12-22; v6 introduces the v3 Language Model Specification). Each stage resolves a `{provider, model, params}` triple to a LanguageModel via the provider packages (`@ai-sdk/anthropic`, `@ai-sdk/openai`, `@ai-sdk/google`, plus `@ai-sdk/openai-compatible` for local/vLLM/Ollama). Typed-JSON stages call `generateObject({ model, schema, prompt })` with a Zod/Standard-Schema schema; free-text stages call `generateText`. A single `LanguageModelV3Middleware` records usage+cost into eleatic on every call. Two viable model-routing topologies: (A) direct provider SDKs (you hold each vendor's API key); (B) a gateway in front -- Vercel AI Gateway (`@ai-sdk/gateway`, model strings like `anthropic/claude-opus-4-8`) or OpenRouter via `@openrouter/ai-sdk-provider` -- so one key reaches all providers and the gateway returns authoritative per-call USD. Recommended: AI SDK 6 as the abstraction + a gateway for prod cost/routing, with direct provider keys as fallback. Confidence: high on AI SDK 6 GA and provider list; medium on exact provider-package version pins (verify in package.json at install time).  
**structuredOutputs:** Parity is now good across the three majors but the mechanism differs, so the abstraction must normalize it. Anthropic: native structured outputs are GA (no beta header) via `output_config.format` ({type:'json_schema', schema}) using constrained decoding (compiled-grammar, cached 24h); the docs list Opus 4.8 / Sonnet 4.6 / Haiku 4.5 among supported models. OpenAI: `response_format` json_schema with `strict:true` gives guaranteed schema adherence (plus a refusal field). Google Gemini: `responseJsonSchema` (full JSON Schema, incl. anyOf/$ref/additionalProperties:false added Nov 2025) or the older OpenAPI-subset `responseSchema` -- strict but historically weaker on deeply-nested schemas. Open/local models (vLLM/Ollama via openai-compatible): json_schema support is best-effort and engine-dependent. The AI SDK papers over the gap with `generateObject`: for Anthropic it exposes a provider option (auto -> native `output_format`, else `jsonTool` tool-call emulation), OpenAI/Gemini map to their native strict modes, and weak providers fall back to tool/JSON-mode emulation. Net: prefer native strict mode where available, keep `generateObject` + a Zod re-validate + bounded repair retry as the uniform safety net so every stage emits validated typed JSON regardless of provider. Caveat (verify): Anthropic structured output historically conflicted with extended thinking in the AI SDK (vercel/ai issues #7220/#9351) -- relevant since your wrapper uses adaptive thinking.  
**costAndUsageCapture:** USD cost is non-standard: no provider returns it natively except gateways. The AI SDK normalizes USAGE only -- `result.usage` exposes inputTokens/outputTokens with `inputTokenDetails` (noCacheTokens, cacheReadTokens) and `outputTokenDetails` (reasoningTokens), plus `usage.raw` for the provider-specific object -- but does NOT compute USD. So the cost layer is a deliberate choice: (1) GATEWAY path -- OpenRouter returns authoritative `usage.cost` + `usage.cost_details` (upstream_inference_cost, cache_discount) inline in every response, and Vercel AI Gateway surfaces per-request cost; use this as cost-of-record when routing through a gateway. (2) SELF-COMPUTED path -- keep your existing `src/llm/pricing.ts` map and extend it to a `Record<provider/model, pricing>` so direct-SDK calls still get USD; this also lets eleatic recompute historical cost deterministically if a number is disputed. Recommendation: a `costUsd` resolver that prefers gateway-reported cost when present and falls back to the local pricing map, writing both the raw provider usage and the resolved USD into the eleatic trace. Note your current cache multipliers (read ~0.1x, 5-min write ~1.25x) are Anthropic-specific and must become per-provider.  
**perStageSwapAndAB:** Model `workflow_version` as a hashable config: `{ [stage]: {provider, model, params} }` (params = effort/thinking/temperature/maxTokens/structured-output mode). Hash the canonicalized (sorted-key) JSON to a stable `workflow_version_id` -- THAT hash is the eleatic arm/run label, so two runs differing only in `code.provider = openai vs anthropic` are distinct comparable arms. This generalizes your existing `STAGE_MODELS` map (currently the per-stage cost lever) from `Record<Stage, ModelId>` to `Record<Stage, {provider, model, params}>`. Because the AI SDK resolves any `{provider,model}` to a uniform LanguageModel and `generateObject` keeps the typed-JSON contract identical across providers, swapping one stage's provider is a config edit with zero call-site change -- the cleanest possible A/B axis. A/B = run the same input under two `workflow_version` hashes and diff eleatic traces on the grounding-gate/critic scores plus the per-stage `costUsd`. Pin params in the config (not in code) so the hash fully determines the arm.  
**claudeceptionFit:** DECISION-CHANGING and the most likely-stale claim to verify: as of 2026-02-19 Anthropic's Legal & Compliance + Consumer Terms EXPLICITLY prohibit routing product/Agent-SDK requests through Free/Pro/Max OAuth credentials -- "Anthropic does not permit third-party developers to ... route requests through Free, Pro, or Max plan credentials," and "Developers building products or services ... including those using the Agent SDK, should use API key authentication."
