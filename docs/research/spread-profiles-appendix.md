# Eval-platform spread — per-platform profiles (with verification)

## Braintrust  
*trust: high* — The leading "code-first eval + production observability + prompt playground" platform: an immutable Eval()/experiment model layered over high-volume trace logging (Brainstore) with online LLM-as-judge scoring, human-review queues, and a Claude-first AI proxy. Production-grade and well-supported, but largely proprietary, score-and-data metered, and self-host gated to Enterprise.

**category:** Hosted LLM eval + observability + experimentation platform (with enterprise hybrid self-host)  
**deployment:** Cloud-first SaaS (api.braintrust.dev / gateway.braintrust.dev). Enterprise tier adds a HYBRID "bring-your-own-cloud" model: customer deploys the DATA PLANE (Braintrust API, PostgreSQL, Redis, object storage/S3, and the Brainstore query engine) into their own AWS/GCP/Azure VPC via official Terraform modules (terraform-aws-braintrust-data-plane) or Helm; Braintrust hosts only the CONTROL PLANE (web UI, auth, non-sensitive metadata like project/org names). Per the docs, sensitive data (logs, traces, spans, datasets, prompt completions) never traverses Braintrust infrastructure — SDKs send directly to the data plane and the browser talks to it via CORS; control/data planes communicate only for auth + metadata sync. This gives real data-residency/data-plane isolation, but ONLY on Enterprise (custom-priced). Not OSS; no free self-host. Data residency posture: strong on Enterprise hybrid; on cloud tiers data lives in Braintrust's environment.  
**tracingModel:** Hybrid: primarily a proprietary SDK (wrapAnthropic / wrapOpenAI, the @braintrust/* SDKs in TS, Python, Ruby, Go, Java, .NET) with a first-class OpenTelemetry BRIDGE rather than being OTel-native at the core. OTLP collector endpoint at https://api.braintrust.dev/otel ingests standard OTel spans; a BraintrustSpanProcessor (TS+Python) sends spans and enables bidirectional parent/child nesting between OTel and Braintrust spans using shared OTel-format trace/span IDs. Implements OpenTelemetry GenAI semantic conventions (maps gen_ai.input.messages->input, gen_ai.completion->output, gen_ai.usage.prompt_tokens->metrics.prompt_tokens) and also accepts a proprietary braintrust.* attribute namespace. Works out-of-the-box with OpenLLMetry, Vercel AI SDK, CrewAI. Net: you can instrument with vendor-neutral OTel and point it at Braintrust, but the richest experience (eval harness, online scoring config) is via their SDK.  
**evals:** Strong, code-first. Eval() function (TS/Python) ties together {data: dataset, task: async fn, scores: [scorers]}; runs produce immutable, comparable, shareable EXPERIMENTS. Open-source autoevals scorer library (Factuality, Battle, etc.) plus custom code scorers and LLM-as-judge scorers. ONLINE scoring runs scorers against live production logs via project-level Automation rules (Logs page) with configurable sampling rate (guidance: 1-10% high volume, 50-100% critical), trace-vs-span scope, and BTQL/SQL WHERE filters (note: != unsupported, use IS NOT); results attach to logs as a child scoring span with a 0-1 or 0-100 value, async, no added app latency. BaseExperiment() supports "hill climbing" (prior output as new expected). Human review: queue-based annotation (Triage / SME / Calibration queues), scores can "write to expected" to build golden datasets, span-level review scores, comments + tags. Datasets are first-class (input/expected/metadata), buildable from scored production traces.  
**experiments:** Offline experiment comparison is the core strength: immutable experiments are stored permanently and diffed side-by-side (metrics, trial variance via trialCount, regression detection) across workflow/prompt versions in the UI and API — directly matching a "workflow VERSION = a run with a baseline" model. This is RUN-COMPARISON, not a hosted ONLINE A/B/assignment+stats engine: Braintrust does not advertise a built-in production traffic-splitting/assignment/statistical-significance experimentation service. Online operation is covered by logging + online scoring + the AI proxy (which can route/compare models), but live randomized experiment assignment with a stats engine would need to be built around it or handled by a separate feature-flag/experimentation tool.  
**claudeSupport:** Excellent — Claude is a first-class citizen. Call Claude from the playground, SDKs, API, and the AI proxy/gateway (gateway.braintrust.dev), including an OpenAI-compatible path so an OpenAI client can target a Claude model name (e.g. claude-sonnet-4-6 / docs reference claude-sonnet-4-5-20250929). wrapAnthropic (and auto-instrumentation import hooks) trace the Anthropic SDK: captures input messages, system prompt, model, params, response content, stop reason/sequence; supports STREAMING; PROMPT CACHING metrics (prompt_cached_tokens, prompt_cache_creation_tokens); server-side TOOL USE (server_tool_use_* counters); structured outputs (output_config); and Anthropic MANAGED AGENTS / sessions (anthropic.beta.agents.*, anthropic.beta.sessions.*). Anthropic publishes a Braintrust customer case study. Strong fit for a Claude-driven ANALYSIS->SYNTHESIS pipeline.  
**pricing:** Three tiers (restructured ~March 16, 2026; pre-March-16 signups grandfathered into features they already created). STARTER (free): 1 GB processed data/mo (+$4/GB overage), 10k scores/mo (+$2.50/1k), 14-day retention, $10/mo Topics(LLM-as-judge) credit, unlimited users/projects/datasets/playgrounds/experiments, Google-only SSO. PRO ($249/mo): 5 GB/mo (+$3/GB), 50k scores/mo (+$1.50/1k), 30-day retention, RBAC, custom charts, priority support; Topics credit promo $249/mo through Sep 1 2026 then $100/mo; Google-only SSO. Topics/LLM-judge metered at $0.06/MTok input + $0.40/MTok output across all plans. ENTERPRISE (custom): self-hosting/hybrid, SAML/OIDC SSO, custom retention/export, SOC2, BAA, SLAs, Slack support. COST METERS = processed data (GB), scores (count), and LLM-judge tokens — NOT per-seat (users unlimited on every tier). For a small-but-growing app, the bill is driven by trace data volume + how aggressively you run online scoring, and the 14-day retention on free tier is tight.  
**bestFitWhen:** You want one hosted platform for live, multi-user operation at scale where Claude is the primary model, you value a mature code-first eval+experiment harness with strong production tracing and online LLM-judge scoring, you can live with a proprietary SDK core (with an OTel escape hatch), and you either accept cloud SaaS or can fund the Enterprise tier to get data-plane isolation in your own VPC. Less ideal if you need free/cheap self-hosting, an open-source core, or a built-in online A/B assignment + stats engine out of the box.  
**eleaticCoexistence:** Clean handoff, low friction. eleatic stays the LOCAL-FIRST/offline/CI layer (single-SQLite store, framework-free explorer); Braintrust becomes the hosted production sink. The shared seam is OpenTelemetry: instrument the ANALYSIS->SYNTHESIS pipeline once with OTel GenAI semconv (or wrapAnthropic) and (a) export OTLP to Braintrust's https://api.braintrust.dev/otel for live ops, while (b) your pipeline still emits the eleatic EvalTrace{spans{kind,input,output,metrics{promptTokens,completionTokens,costUsd,durationMs}}} for offline. The data models line up well: eleatic eval_run<->Braintrust experiment (both version-a-run-with-baseline), eval_row<->log/experiment row (opaque input/expected/output/scores/metadata), span metrics<->OTel gen_ai.usage.* / metrics.*; eleatic's pure analysis (auc/calibratedThreshold/labelAgreement) and adjudication overlap with Braintrust autoevals + human-review queues, so you can keep AUC/calibration math in eleatic and use Braintrust queues for live human review. Migration/exit cost is moderate: experiments/datasets/logs are exportable (export is an Enterprise feature for automation), but BTQL/Brainstore/autoevals config and online-scoring rules are Braintrust-specific and would not transfer. Recommended split: eleatic = source of truth for offline/CI gating and portable traces; Braintrust = live telemetry + online scoring + human review, fed by the same OTel-instrumented trace.  

**Strengths:** Mature, code-first eval harness (Eval/task/scorers) + open-source autoevals; immutable, permanently-comparable experiments map cleanly onto a 'workflow version = run with baseline' model; Production observability built for scale via Brainstore (Rust query engine over Postgres/Redis/object storage) for fast search/filter over deeply nested AI traces; Online LLM-as-judge scoring on live logs with sampling + BTQL filters, async and off the request latency path; Best-in-class Claude support: playground/SDK/proxy, deep Anthropic tracing (caching, tool use, managed agents, streaming), OpenAI-compatible gateway, Anthropic case study; Real enterprise hybrid self-host: data plane in your own VPC via Terraform/Helm, sensitive data never crosses Braintrust servers — genuine data-residency isolation; Robust human-review workflow (triage/SME/calibration queues, write-to-expected to grow golden datasets); No per-seat pricing — unlimited users at every tier; OTel bridge with GenAI semantic conventions eases interop and reduces total lock-in

**Weaknesses:** Largely proprietary and not OSS — the richest features assume the Braintrust SDK; OTel is a bridge, not the native core, so deep usage creates lock-in (Brainstore, BTQL, autoevals config, prompt store); Self-hosting / data-plane isolation, SAML/OIDC, custom retention, BAA, SLAs are ALL Enterprise-only (custom-priced) — no affordable self-host or free data residency; Cost meters on DATA volume + score count + judge tokens; high-traffic tracing plus aggressive online scoring can scale costs non-linearly, and free-tier retention is only 14 days (Pro 30 days); No built-in ONLINE A/B assignment + statistical-significance experimentation engine; it's offline run-comparison plus logging — live experiment assignment must be built or bolted on; Pricing was restructured in March 2026 (Starter replaced Free; Topics promo pricing reverts Sep 1 2026) — numbers are in flux and should be re-verified; LLM-as-judge online scoring inherits judge cost/latency/bias; meaningful coverage at high volume forces low sampling rates

## Langfuse  
*trust: medium* — The leading OSS-first LLM observability/eval stack: MIT-licensed core you can fully self-host (tracing, evals, prompt mgmt, datasets, dashboards) with a small set of EE features gated behind a license key — the main reason to prefer it over a closed SaaS like Braintrust. As of Jan 2026 it's owned by ClickHouse.

**category:** Open-source-first LLM observability + evaluation + prompt-management platform (cloud or self-hosted). Now a ClickHouse company (acquired Jan 2026).  
**deployment:** All three: (1) Langfuse Cloud (hosted SaaS, EU + US data regions selectable at signup); (2) self-host OSS (Docker Compose / Helm-on-k8s; AWS Marketplace AMI available) under MIT with no feature/volume limits on core; (3) self-host commercial (Pro self-serve key or Enterprise contract) which unlocks the /ee modules. Same container image across all three — you switch posture by adding/removing LANGFUSE_EE_LICENSE_KEY at runtime, so OSS->cloud or cloud->self-host migration is low-friction. Stack components: Langfuse web/worker + Postgres (transactional) + ClickHouse (analytics/traces, mandatory since v3) + Redis + S3/blob. Data residency: self-host = your infra; cloud = pick EU or US. Note: ClickHouse Inc. acquired Langfuse in Jan 2026 — deployment model is unchanged today but is a watch-item.  
**tracingModel:** OpenTelemetry-native (the strongest OTel story among the eval-platform peers). SDK v4 is "a thin layer on top of the official OpenTelemetry client" — emitted spans are standard OTel spans. Langfuse exposes an OTLP ingestion endpoint at /api/public/otel (OTLP over HTTP/JSON and HTTP/protobuf; gRPC not yet supported) with Basic Auth. It maps incoming traces from the OpenTelemetry GenAI semantic conventions AND OpenInference-style attributes (input.value/output.value), so it can act as a backend for any OTel/OpenInference instrumentation (OpenLLMetry, openinference instrumentors, OTel Collector pipelines). Also has native Python/JS SDKs (decorators/context managers) which are recommended over raw OTel for those two languages. Proprietary additions exist (sessions, scores, user_id/tags) layered on the OTel data model.  
**evals:** Full eval suite, but the higher-value evaluators are EE-gated even on self-host (see weaknesses). Methods: LLM-as-a-judge (managed evaluators, run online on production traces or offline on datasets), code/custom evaluators, API/SDK custom scoring, and human review via Annotation Queues (incl. open-ended text scores). Datasets = reusable test-case collections; Experiments compare versions side-by-side offline or online; CI/CD integration can block deploys on regressions. Score Analytics dashboards trend quality/tonality/accuracy/completeness over time. Scores are first-class on the trace data model. Prompt management (versioning, labels, deployment) and an LLM Playground round it out.  
**experiments:** Primarily offline run-comparison: Datasets x prompt/model versions run as "Experiments" with side-by-side diffing and regression gating in CI. Online evaluation exists (LLM-judge scoring live production traces). It does NOT ship a built-in online A/B traffic-assignment + statistical-significance engine the way a dedicated experimentation product would — "experiments" here means dataset-driven eval runs, not randomized live traffic splits with a stats backend. For true online A/B you'd instrument assignment yourself and use scores/dashboards to compare cohorts.  
**claudeSupport:** Solid but not "native first-party SDK." Two documented paths: (1) OpenTelemetry instrumentation via AnthropicInstrumentor (openinference/OpenLLMetry-style) wrapping the Anthropic SDK and shipping OTel spans to Langfuse; (2) Anthropic's OpenAI-compatible endpoint behind Langfuse's OpenAI SDK wrapper. Captures prompt/response, token usage, latency, cost, plus custom attrs (user_id, session_id, tags, version). Model/cost catalog includes Anthropic models for cost tracking. There is no Anthropic-specific drop-in wrapper equivalent to the OpenAI one — Claude goes through the OTel/instrumentation route, which is fine given the OTel-native design.  
**pricing:** CLOUD (4 tiers; "unit" = any tracing data point — trace, observation/span/event/generation, or eval score): Hobby free (50k units/mo, 30-day retention, 2 seats, community support); Core $29/mo (100k units, 90-day retention, unlimited users, in-app support); Pro $199/mo (100k units, 3-year retention, unlimited users, SOC2/ISO27001, data-retention mgmt, Teams add-on); Enterprise $2,499/mo (audit logs, SCIM, dedicated engineer, custom SLAs). Graduated overage on all tiers: $8/100k (100k-1M), $7/100k (1M-10M), $6.50/100k (10M-50M), $6/100k (50M+). Example: 5M units/mo on Pro ~= $551 total. SELF-HOST: OSS core is free/unlimited under MIT (no key). A self-serve "Pro" license key unlocks EE features for small teams (pay-as-you-go; exact $ not clearly published on the vendor page — verify) and Enterprise is custom-quoted and explicitly "additive to your ClickHouse commercial plan." COST REALITY of self-host: free license != free to run — you operate Postgres + ClickHouse + Redis + S3/blob + web/worker; ClickHouse + storage are the real cost/ops burden, especially at trace volume.  
**bestFitWhen:** You want production LLM observability + evals but require an OSS, self-hostable, OTel-portable foundation (data-residency/compliance, no vendor lock-in, ability to run in your own VPC) — and you're willing to either run ClickHouse yourself or pay for a license/cloud for the gated eval/review features. Strong fit when "we may need to bring this in-house / keep data on our infra" is a hard requirement, which is precisely the axis where it beats Braintrust. Less ideal if you specifically need turnkey online A/B experimentation or want zero infra ops with the richest managed eval UX out of the box.  
**eleaticCoexistence:** Clean coexistence and a natural handoff. Because Langfuse is OpenTelemetry-native (OTLP endpoint + GenAI semconv/OpenInference mapping), the SAME instrumented agent pipeline that emits eleatic's EvalTrace{spans} can ALSO emit OTel spans to Langfuse — one instrumentation layer, two sinks. Mapping is direct: eleatic span.kind -> OTel span / Langfuse observation; span.metrics{promptTokens, completionTokens, costUsd, durationMs} -> OTel GenAI token/cost/latency attributes (Langfuse already computes cost/latency); an eleatic eval_run (workflow version + baseline) -> a Langfuse dataset-run/experiment or a release/version tag; eval_row scores -> Langfuse scores; eleatic adjudication/human review -> Langfuse Annotation Queues (EE). Recommended division of labor: eleatic stays the local-first, offline, single-SQLite eval store for CI/pre-ship gating and the pure analysis module (auc/calibratedThreshold/labelAgreement); Langfuse is the hosted, persistent, multi-user PRODUCTION telemetry sink + online LLM-judge scoring + dashboards at scale. Handoff cost is low precisely because both sit on the OTel data model — you can fan-out from a single OTel SDK (Langfuse via OTLP, eleatic via its own exporter) or batch-replay production traces back into eleatic's offline analyzer. This OTel bridge is the same reason Langfuse migrations are cheap and is a strong argument for the eleatic-offline + Langfuse-production pairing over a closed-SaaS pairing.  

**Strengths:** Genuinely OSS-first: MIT-licensed core with tracing, evals, prompt management, datasets, dashboards, and full API available self-hosted with NO volume or feature caps — the cleanest open-core story versus closed SaaS like Braintrust.; Best-in-class OpenTelemetry posture: real OTLP endpoint, GenAI semconv + OpenInference mapping, SDK is a thin layer over the OTel client — so instrumentation is portable and not lock-in.; Identical image across OSS/self-host-EE/cloud; flip a license key to move between them, so you can start offline/self-hosted and graduate to cloud (or vice versa) cheaply.; Cloud pricing is usage-based with UNLIMITED seats on paid tiers — doesn't tax engineering headcount the way per-seat competitors do.; Mature, popular, large community; broad framework integrations (LangChain, LlamaIndex, OpenAI/Anthropic instrumentors, etc.).; Now backed by ClickHouse's resources/balance sheet (Jan 2026 acquisition), reducing startup-mortality risk.

**Weaknesses:** Several high-value features are EE/commercial-gated EVEN on self-host: LLM-as-a-judge evaluators, LLM Playground, human Annotation Queues, and Prompt Experiments are reported to require a paid commercial plan/license, alongside RBAC, audit logs, data masking, SCIM, retention policies. The free OSS tier is tracing + prompt mgmt + datasets + dashboards, not the full eval/review workflow. (Verify exact OSS-vs-EE split against current docs — this is the most fluid area.); Self-hosting is operationally heavy: mandatory ClickHouse (v3+) plus Postgres, Redis, and blob storage — not a single-binary deploy. Real infra + ops cost.; No native randomized online A/B experimentation engine with statistical significance — 'experiments' are dataset/offline-oriented; live A/B is DIY.; ClickHouse acquisition (Jan 2026) is recent — long-term independence of the OSS roadmap, pricing, and whether the managed offering steers toward ClickHouse Cloud are open questions.; Claude support is via OTel/instrumentation, not a first-party Anthropic wrapper.

## Arize Phoenix / Arize AX  
*trust: high* — The strongest "avoid lock-in via open standards" pick: OSS Phoenix gives full tracing/evals/experiments under Elastic License 2.0, instrumented via Apache-2.0 OpenInference on top of OpenTelemetry, so the same trace can flow to Phoenix, Arize AX, or any OTLP backend; AX is the hosted/enterprise upgrade with online evals, SLAs, and compliance.

**category:** OpenTelemetry/OpenInference-native LLM tracing + evals + experiments. Two products: Phoenix (OSS, self-hostable, ELv2) and Arize AX (commercial hosted + enterprise self-host) for production-scale observability, online evals, and experiments.  
**deployment:** Hybrid. Phoenix OSS self-hosts via Docker, Docker Compose, Kubernetes, Helm, AWS CloudFormation, Railway, or local/notebook; backs onto SQLite or PostgreSQL; data never leaves your infra by default. There is also a managed "Phoenix Cloud" (app.phoenix.arize.com). Arize AX is primarily SaaS cloud (Free/Pro), with self-host + data residency + multi-region only on the Enterprise tier. Caveat (low confidence): exact AX self-host packaging/regions are documented at a high level on the pricing page and should be confirmed with sales.  
**tracingModel:** OpenTelemetry-native. Phoenix ingests traces over OTLP and is "vendor, language, and framework agnostic." Instrumentation is via OpenInference (Apache-2.0), a set of semantic conventions + auto-instrumentors complementary to OTel covering 30+ frameworks/providers across Python, JS/TS, Java, Go. OpenInference is its own AI-span convention (not the same as OTel's GenAI semconv, though both layer on OTel) — Arize maintains it as the reference implementation. Critically, OpenInference traces "can be used with any OpenTelemetry-compatible backend," not just Phoenix/Arize, which is the basis of the avoid-lock-in claim. AX additionally ingests OpenLLMetry/OTel.  
**evals:** Strong. Phoenix ships an evals library (arize-phoenix-evals) with LLM-as-judge evaluators (Hallucination, QA, Relevance, Toxicity, Summarization, retrieval relevance), plus code-based checks and human annotation/labeling. Evaluators can use Claude models as the judge. Datasets are versioned; experiments track changes to prompts/models/retrieval. AX adds online evals (continuous/streaming scoring of production traces) and "Alyx" agent-assisted analysis. Offline eval = run evaluators over datasets/traces in Phoenix; this is the natural overlap with eleatic.  
**experiments:** Offline experiment/run-comparison is first-class in both Phoenix and AX (versioned datasets + experiments to compare prompt/model/retrieval variants side-by-side with scores). Online evals exist in AX (scoring live production spans), but there is no documented built-in online A/B traffic-splitting / assignment / statistical-significance engine the way a dedicated experimentation platform offers — experimentation here is dataset-and-eval-driven, not randomized-traffic-driven. Treat "online A/B with stats engine" as not a core strength (medium confidence).  
**claudeSupport:** First-class. OpenInference provides auto-instrumentation for the Anthropic SDK and the Claude Agent SDK (openinference-instrumentation-anthropic is the reference implementation), with documented Claude Code / Agent SDK tracing flows into both self-hosted Phoenix and AX. Claude models are also supported as LLM-judge evaluators in phoenix.evals. So Claude is covered both as the traced subject and as the evaluator.  
**pricing:** Four tiers (verified on arize.com/pricing, cross-checked May 2026 third-party sources — treat exact numbers as the most likely-to-drift facts). Phoenix OSS: free, no usage/retention caps, self-hosted, full-featured. AX Free: $0, ~25k trace spans/mo, 1 GB/mo ingestion, 15-day retention, includes online evals + Alyx, community support, cloud-only. AX Pro: $50/mo, ~50k spans/mo, 10 GB/mo, 30-day retention, email support, cloud-only. AX Enterprise: custom (third-party sources cite a ~$60k/yr median, low confidence) — adds self-host, data residency/multi-region, SLA, SOC2 Type II + HIPAA + GDPR, SSO/RBAC, audit logs, dedicated support. Startup discount available on application. Cost meter = trace spans + GB ingested + retention window; seats not the primary meter.  
**bestFitWhen:** You want production observability + online evals at scale but refuse hard vendor lock-in: instrument once with OpenInference/OTel, self-host Phoenix OSS early (keeping topic-synthesis prompts/PII in-house at zero cost), and graduate to Arize AX only if/when you need hosted online evals, SLAs, and SOC2/HIPAA. Especially apt because the agent pipeline is Claude-based (first-class instrumentation + Claude-as-judge) and because the OTLP exit ramp de-risks the whole bet.  
**eleaticCoexistence:** Strong, low-friction coexistence — best-in-class for this pairing. Because Phoenix/AX consume OTLP via OpenInference, the topic-synthesis agent pipeline can emit ONE OpenTelemetry trace per workflow run and fan it out: export to Phoenix/AX (production telemetry) while a thin adapter projects the same spans into eleatic's EvalTrace{spans:[{kind,input,output,metrics{promptTokens,completionTokens,costUsd,durationMs}}]} for the offline SQLite store. OpenInference span attributes (token counts, model, cost, latency, input/output) map almost directly onto eleatic's span metrics, so the handoff is a field-mapping, not a re-instrumentation. Division of labor: eleatic = offline/CI single-file eval-results store + framework-free explorer with pure analysis (auc/calibratedThreshold/labelAgreement) and adjudication; Phoenix/AX = live persistent telemetry + online evals at scale. Each generated artifact becomes an eleatic eval_row AND a Phoenix-traced run. Handoff cost is minimal and reversible: because instrumentation is OTLP/OpenInference (not a proprietary SDK), you are never locked out of feeding eleatic, and you can drop Phoenix/AX without re-instrumenting. Braintrust's "experiments" are offline eval-run comparisons over datasets, NOT randomized online A/B — so if you want statistically-valid live A/B of whole workflows you need one of these alongside it. For topic-synthesis at launch you almost certainly do NOT need it yet.

## Interop / lock-in / OpenTelemetry standards  
*trust: high* — Instrument the multi-agent pipeline ONCE as OpenTelemetry GenAI spans (via OpenLLMetry or hand-rolled gen_ai.* attributes), fan out over OTLP to a production backend, and convert the same span tree into eleatic's opaque { spans } blob with a ~1-file adapter — OTel is the de-facto interop layer even though its GenAI semconv is still officially "Development."

**category:** Cross-cutting standards layer (instrumentation wire format), not a product. Covers OpenTelemetry GenAI semantic conventions + the two competing OTel-compatible AI instrumentation libraries (OpenLLMetry/Traceloop, OpenInference/Arize), and how each candidate backend (Braintrust, Langfuse, Arize Phoenix) ingests them.  
**deployment:** Standard itself is vendor-neutral OSS (Apache-2.0 spec + SDKs). OTLP/HTTP and OTLP/gRPC are the universal wire transport. Ingest posture differs by backend: Braintrust = cloud (US/EU regions) + self-host, converts OTLP into a PROPRIETARY internal store; Langfuse = OSS self-host or cloud, OTLP feeds the SAME ClickHouse pipeline as native SDK (closest to OTel-native); Arize Phoenix = OSS self-host (single node) or Arize cloud, OTel-native over OTLP gRPC/HTTP, owns OpenInference. eleatic = local single-SQLite file, no network, stores trace as an opaque JSON blob.  
**tracingModel:** OpenTelemetry-native is the interop substrate. Three overlapping flavors: (1) OTel GenAI semantic conventions — official spec, status DEVELOPMENT (not Stable) as of v1.38.0 / mid-2026; defines gen_ai.operation.name {chat, execute_tool, invoke_agent, embeddings, create_agent}, gen_ai.provider.name {anthropic, openai, aws.bedrock, ...}, gen_ai.usage.input_tokens / output_tokens, span names like "chat {model}", "execute_tool {tool}", "invoke_agent {agent}". (2) OpenLLMetry (Traceloop) — OTel-based auto-instrumentation, broad provider coverage incl. Anthropic; emits gen_ai.* and historically gen_ai.prompt/gen_ai.completion (now deprecated, migrating to message-structured + log events). (3) OpenInference (Arize) — OTel-aligned but a SEPARATE attribute namespace (openinference.*/llm.*), drop-in per-package instrumentors, reference backend is Phoenix. All three ride OTLP, so the transport is portable; the ATTRIBUTE schema is where flavor lock-in hides. Notable gap: cost (USD) is NOT a defined OTel GenAI attribute — only token counts are; backends compute cost from pricing tables.  
**evals:** N/A at the standards layer — OTel/OpenLLMetry/OpenInference only standardize TRACES (and some metrics/logs), not eval verdicts, scorers, or datasets. There is no cross-vendor standard for an "eval row" or "score." This is precisely the seam where eleatic (offline scores/auc/adjudication) and a production backend (Braintrust online scorers, Phoenix evals, Langfuse evals) each keep their own proprietary eval model. Instrumentation portability does NOT buy eval portability.  
**experiments:** N/A at the standards layer. OTel has no concept of an experiment, A/B assignment, or variant. Experiment/assignment semantics are 100% backend-proprietary. The only portable hook is putting a workflow-version / variant id into a span attribute (e.g. a custom attribute or gen_ai.* metadata) so every backend AND eleatic can group by it — that convention is yours to define and carry.  
**claudeSupport:** Strong for Anthropic/Claude. gen_ai.provider.name defines "anthropic" as a first-class value. OpenLLMetry ships Anthropic auto-instrumentation (one of its more mature integrations). For a Claude-Agent-SDK / multi-agent pipeline, invoke_agent and execute_tool operation names map the agent + tool-call structure natively. Caveat: the Claude SDK does not emit OTel GenAI spans out of the box in all paths; you typically add OpenLLMetry's Anthropic instrumentor or hand-set gen_ai.* attributes on your own spans. Cached-token accounting (prompt_cached_tokens) maps cleanly into Braintrust metrics; OTel covers cached tokens within input_tokens.  
**pricing:** The standard is free/OSS (Apache-2.0). Cost is incurred at the backend, metered there (Braintrust meters by spans/events ingested + seats; Langfuse OSS self-host is free, cloud meters by events/units; Phoenix OSS self-host free, Arize cloud is seat/volume). Instrumenting once with OTel does NOT change a backend's meter — it just lets you switch which meter you feed without re-instrumenting. LOW-CONFIDENCE on exact backend tiers/numbers; not verified in this pass (out of scope for the standards target — see the per-vendor profiles).  
**bestFitWhen:** You want to commit to Braintrust for production WITHOUT betting the instrumentation on it. Best fit: emit OTel GenAI (gen_ai.*) spans — either via OpenLLMetry's Anthropic instrumentor or by hand-setting gen_ai.* on your own agent/tool spans — register a multiplexing OTLP SpanProcessor that (a) exports to Braintrust's OTLP endpoint and (b) hands the finished span tree to a small in-process eleatic adapter. Avoid OpenInference unless you intend to standardize on Arize/Phoenix. Avoid raw vendor SDK span formats (incl. Braintrust's native proprietary span format) as your PRIMARY emission path, since that re-introduces the lock-in you're trying to dodge.  
**eleaticCoexistence:** CENTRAL ANSWER: YES — one instrumentation feeds both, and the eleatic adapter is cheap. eleatic's EvalRowRecord.trace is an OPAQUE JSON blob; its house shape is { spans: [{ name, input, output, usage }] } but anything round-trips. So the production path and the offline path SHARE one set of OTel spans. Concrete coexistence design: (1) Instrument the pipeline once with OTel GenAI semconv (gen_ai.operation.name = invoke_agent/chat/execute_tool; gen_ai.usage.input_tokens/output_tokens; gen_ai.provider.name=anthropic). (2) Production fan-out: a BatchSpanProcessor exports OTLP to Braintrust (api.braintrust.dev/otel/v1/traces with Authorization + x-bt-parent headers); Braintrust auto-maps gen_ai.prompt->input, gen_ai.completion->output, gen_ai.request.*->metadata, gen_ai.usage.*->metrics and computes estimated cost from its pricing table. (3) Offline fan-out: a second in-process SpanExporter collects the same finished spans per workflow run and a ~80-150 LOC adapter maps each OTel span -> eleatic span { name: span.name, kind: gen_ai.operation.name, input: gen_ai input messages, output: gen_ai output, metrics: { promptTokens: gen_ai.usage.input_tokens, completionTokens: gen_ai.usage.output_tokens, durationMs: end-start, costUsd: (computed locally from a pricing map, since OTel has no cost attr) } }, then writes one EvalRowRecord per generated artifact (output/expected/scores) with trace={spans:[...]}. ADAPTER COST ESTIMATE: low — roughly one file, 1-2 days. The hard parts are (a) injecting costUsd locally because OTel omits it, (b) flattening the OTel parent/child span tree into eleatic's flat spans array while preserving order, and (c) deciding a stable rowKey (e.g. concept-page id) so offline diff/adjudication lines up across workflow versions. HANDOFF/SYNC: the workflow-version id should be a single custom span attribute carried on the root span; Braintrust groups experiments by it and eleatic uses it as the eval_run id/baseline — keeping offline runs and online traces addressable by the SAME version key. MIGRATION COST if you later drop Braintrust: near zero for instrumentation (re-point the OTLP exporter); eval/experiment history in Braintrust does NOT come back out as OTel, so that data is the lock-in, not the traces.
