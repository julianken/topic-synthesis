# Discovery — per-dimension findings (with verification)

## Competitive & prior-art landscape  
*overall trust: high*

The market splits into five clusters, none of which delivers topic-synthesis's exact value proposition — auto-generated, interactive, visual, scaffolded curricula from any topic. (1) AI tutors/copilots (Khanmigo, Coursera Coach, Duolingo Max, Sana, Synthesis, Brilliant) are conversational and adaptive but bolt onto pre-authored, human-curated content; they answer questions about a fixed curriculum rather than generating a new one per topic. (2) Interactive-explainer platforms (Mathigon, Nicky Case's Explorable Explanations, 3Blue1Brown) embody exactly the pedagogy and artifact format topic-synthesis wants — hand-built, manipulable, visual, scaffolded — but are entirely human-authored and do not scale to arbitrary topics. (3) Notebook/synthesis tools (NotebookLM) generate study guides, mind maps, flashcards, and audio/video overviews from user-supplied sources, but output is summarization-shaped, not an interactive coded curriculum, and is bounded to documents the user uploads. (4) Generative course/roadmap builders (roadmap.sh, Teachable, Coursebox, AcademyOcean, Coursera Course Builder) do generate structure from a topic, but produce text outlines, slides, and quizzes — not interactive HTML/Canvas/SVG visualizations — and target instructors/L&D, not self-directed learners. (5) A newly emerged primitive — LLMs that emit interactive HTML/SVG/JS (Claude's interactive visualizations, launched March 2026) and Manim-from-prompt video tools — proves the generation capability exists but has only been demonstrated one-explainer-at-a-time, with documented accuracy/fabrication failures and no scaffolding, status tracking, or curriculum-hub structure. The white space is precise: orchestrated generation of a TIERED, status-tracked SITEMAP of interactive explainer pages (the "ai-concept-viz" artifact) from any topic, combining the explorable-explanation pedagogy of cluster 2 with the from-scratch generation of cluster 4/5 — a combination no incumbent currently ships. The central risk is the same one that sinks the Manim-from-prompt demos: AI-generated educational content fabricates concepts and renders flawed visuals, and matching hand-built quality requires expert-level verification — which is exactly why topic-synthesis's eval/A-B-testing requirements are load-bearing, not optional.

### Recommendations
- Position topic-synthesis explicitly as 'Explorable Explanations / Mathigon, auto-generated for any topic' — own the intersection (from-scratch generation + interactive-visual-scaffolded artifacts) that no incumbent occupies. Do NOT position as a tutor/chatbot (Khanmigo/Coursera/Sana own that) or a text-outline generator (roadmap.sh/Coursebox commoditizing it).
- Make the defensible artifact the tiered, status-tracked SITEMAP of interactive explainer pages (the ai-concept-viz hub), not any single page. Single-page generation is already a near-commodity (Claude Artifacts); the orchestration, cross-page coherence, scaffolding gates, and 'built vs soon' status are the moat.
- Architect SYNTHESIS agents to emit standalone HTML/Canvas/SVG/JS artifacts in the ai-concept-viz format, and reuse the proven Claude-Artifacts-style generation primitive rather than reinventing it; spend engineering budget on the analysis→sitemap→multi-page coordination instead.
- Encode concrete pedagogy patterns mined from cluster 2 into DESIGN.md and the synthesis prompts: progressive disclosure / gated sections (Mathigon), isolate-mechanics-then-combine and question-before-answer (Nicky Case), manipulable variables with live feedback (Claude viz), spaced practice/flashcards and mind-map navigation (NotebookLM, Brilliant).
- Treat ANALYSIS-quality and SYNTHESIS-quality evals as first-class, blocking workflow stages targeting the documented failure categories: fabricated/non-existent concepts, factually wrong claims, visuals that don't match the concept, premature/incomplete explanations, and scaffolding soundness. This is the only thing separating the product from the discredited Manim-from-prompt demos.
- Instrument output-side engagement and learning-outcome telemetry from day one and build workflow-level A/B testing in early — this is a genuine differentiator because even Khanmigo lacks RCT efficacy evidence; 'we can show which workflow produces measurably better learning' is a defensible claim.
- Decide the pricing/segment and per-generation cost model early: from-scratch interactive curricula cost far more compute than chat tutoring yet face $4/mo and free anchors. Lean prosumer/B2B or constrain cost via caching, templated visualization components, and generate-on-demand 'soon' tiers.
- Borrow the multi-artifact, navigable UX bar set by NotebookLM's Studio panel and roadmap.sh's flowchart hub for the front-end (built with the transitions-dev library), since users now expect a polished, explorable hub rather than a wall of text.

### Open questions
- What is the unit of generation and caching — a whole curriculum per topic, or individually-cached/reusable concept pages shared across topics (which would slash cost and enable a growing library)?
- Which subject domains generate reliably? Math/CS/physics map cleanly to manipulable visualizations (Mathigon, 3B1B territory); do humanities/soft topics degrade into text, eroding the interactive differentiation?
- What is the verification mechanism for generated visuals specifically (not just text)? Self-critique by another agent, a render-and-screenshot eval, retrieval-grounded checks, or human-in-the-loop for a 'verified' tier?
- Who is the buyer/user — self-directed adult learners, students, teachers/instructional designers, or L&D — and does that segment justify the higher per-output cost vs. the $4/mo–free anchors?
- How does scaffolding get determined automatically — i.e., what produces the tier→category→concept dependency ordering, and how is it validated as pedagogically sound rather than just a topic taxonomy?
- What does 'soon' (unbuilt) status mean operationally — lazy generate-on-demand to control cost, or a curation signal — and how does that interact with the A/B-testing of whole workflows?
- Is there IP/defensibility risk given Claude Artifacts already does single interactive explainers — what stops the underlying model vendor from shipping the curriculum-hub layer natively?

## Learning science & instructional design (defines "good")  
*overall trust: high*

The learning-science literature converges on a small set of robust, replicated principles that can be translated into concrete, machine-checkable properties for a generated artifact. They fall into three buckets relevant to topic-synthesis: (1) Don't overload working memory — Cognitive Load Theory and Mayer's multimedia principles dictate how words, visuals, and interactivity should be composed on each page (high-confidence, large effect sizes, e.g. coherence g≈0.86). (2) Force effortful processing — retrieval practice, spacing, interleaving, self-explanation, and the ICAP "do something generative" hierarchy are the highest-utility techniques (Dunlosky's top tier; self-explanation g≈0.55), and they imply every page must embed active tasks, not just exposition. (3) Sequence and scaffold — Bloom's two-dimensional taxonomy gives measurable objectives, prerequisite/knowledge-graph sequencing orders the curriculum, and ZPD/scaffolding with faded worked examples calibrates difficulty to the learner's stated prior knowledge. Critically, the right design is contingent on the learner: the expertise-reversal effect means guidance that helps novices actively harms experts, so the SETTINGS (prior knowledge, depth) must drive generation, and these principles must become EVAL CRITERIA, not just generation hints. The product's interactive-visualization format is itself well-justified (multimedia + dual-coding + segmenting principles), provided visuals are integrated and non-redundant rather than decorative. Below, each principle is stated as a concrete checkable property that doubles as a synthesis target and an eval rubric item.

### Key findings

- **Cognitive Load Theory: every page must minimize extraneous load and keep intrinsic load matched to the learner's prior knowledge, because working memory is the bottleneck for building long-term schemas.** _(confidence: high)_  
  Sweller's CLT distinguishes intrinsic (task complexity × element interactivity), extraneous (poor presentation), and germane (schema-building) load. Intrinsic load is NOT fixed — it depends on the interaction between task structure and the learner's prior knowledge, so the same material is high-load for a novice and low-load for an expert. The 2023 British Journal of Educational Psychology special issue (Sweller's discussion) reaffirms the framework and its instructional effects.  
  → *so what:* Make CLT operational as eval criteria: (a) cap new element-interactivity per page (chunk via the Segmenting principle); (b) sequence high-interactivity material only after prerequisites; (c) drive a 'novice vs. expert' generation branch off the SETTINGS prior-knowledge slider; (d) an eval can score each generated page for extraneous-load violations (decorative elements, tangents, wall-of-text). Element-interactivity count and per-page new-concept count are concrete metrics.  
  [Discussion of the special issue on cognitive load theory — Sweller, 2023, British Journal of Educational Psychology](https://bpspsychub.onlinelibrary.wiley.com/doi/10.1111/bjep.12606); [Cognitive Load Theory (John Sweller) — InstructionalDesign.org](https://www.instructionaldesign.org/theories/cognitive-load/); [Cognitive Load Theory and individual differences — ScienceDirect, 2024](https://www.sciencedirect.com/science/article/pii/S1041608024000165)
- **Mayer's multimedia principles give ~15 directly checkable composition rules for any page that combines words and visuals — the exact format topic-synthesis generates.** _(confidence: high)_  
  Three decades and 200+ experimental comparisons yield evidence-based principles, several with large effect sizes: Coherence (exclude extraneous material) was supported in 23/23 tests with median d≈0.86; Signaling (cue key material), Redundancy (don't duplicate narration as on-screen text), Spatial & Temporal Contiguity (place/time related words and pictures together), Segmenting (learner-paced chunks), Pre-training (teach key terms first), Modality, Multimedia, Personalization, Voice, and Image principles round out the set.  
  → *so what:* This is the single richest source of mechanical eval rules for the visual pages. Encode each as a lint check on generated HTML/SVG: labels adjacent to the diagram parts they describe (spatial contiguity); no decorative imagery or off-topic asides (coherence); key elements visually signaled; a 'key terms' pre-training block precedes complex visualizations; interactive steps are user-paced (segmenting). Coherence and signaling should be hard gates because their effects are largest.  
  [Evidence-based principles for how to design effective instructional videos — Mayer, 2021, JARMAC](https://www.sciencedirect.com/science/article/abs/pii/S2211368121000231); [Principles for Reducing Extraneous Processing in Multimedia Learning (Coherence, Signaling, Redundancy, Spatial/Temporal Contiguity) — Mayer & Fiorella](https://www.researchgate.net/publication/262915119_Principles_for_reducing_extraneous_processing_in_multimedia_learning_Coherence_signaling_redundancy_spatial_contiguity_and_temporal_contiguity_principles); [Mayer's 12 Principles of Multimedia Learning — Digital Learning Institute](https://www.digitallearninginstitute.com/blog/mayers-principles-multimedia-learning)
- **Retrieval practice (self-testing) and distributed/spaced practice are the two HIGHEST-utility learning techniques; every generated unit should embed retrieval and be spaceable.** _(confidence: high)_  
  Dunlosky et al. (2013, Psychological Science in the Public Interest) rated 10 techniques: practice testing and distributed practice earned 'high utility' (broad benefit across ages, abilities, materials, and real educational contexts). Summarization, highlighting, keyword mnemonics, imagery-for-text, and rereading earned 'low utility.' Cepeda et al. (2006) found spaced practice beat massed in 259/271 comparisons.  
  → *so what:* A page that is pure exposition is, by this evidence, a low-quality artifact. Synthesis target: every bite-size topic page ends with retrieval questions / a self-quiz / an interactive 'predict-then-reveal' step, and the hub supports spaced review (return-later scheduling). EVAL: score presence + quality of retrieval opportunities per page, and flag the low-utility patterns (e.g. the page is just a summary to reread). This directly serves the founder's 'output-side engagement/outcomes' data goal — quiz attempts are measurable signal.  
  [Improving Students' Learning With Effective Learning Techniques — Dunlosky, Rawson, Marsh, Nathan, Willingham, 2013](https://journals.sagepub.com/doi/abs/10.1177/1529100612453266); [Strengthening the Student Toolbox — Dunlosky, American Educator (accessible summary)](https://www.aft.org/ae/fall2013/dunlosky); [Testing effect — Wikipedia (overview with primary-literature citations)](https://en.wikipedia.org/wiki/Testing_effect)
- **Worked examples with faded guidance beat unguided problem-solving for novices — but the benefit reverses for experts (expertise-reversal effect), so guidance level must be a function of the learner's prior knowledge.** _(confidence: high)_  
  Sweller & Cooper (1985) showed studying worked examples outperforms unguided practice by cutting extraneous load. Kalyuga's expertise-reversal effect shows that as expertise grows, worked examples lose and then reverse their advantage; the remedy is backward fading — present a full worked example, then omit the last step, then the last two, transitioning the learner to independent problem-solving.  
  → *so what:* Generate a worked-example-to-practice gradient per skill, and gate it on the SETTINGS prior-knowledge level: novices get full worked examples + faded steps; advanced learners get problems with minimal scaffolding. This is also a concrete A/B-testable workflow variable (faded vs. fixed guidance). EVAL must check that guidance level matches the declared learner level — a generic 'one size' page should be penalized.  
  [Worked-example effect — Wikipedia (Sweller & Cooper 1985 and related)](https://en.wikipedia.org/wiki/Worked-example_effect); [The Expertise Reversal Effect and Its Implications for Learner-Tailored Instruction — Kalyuga et al.](https://www.uky.edu/~gmswan3/EDC608/Kalyuga2007_Article_ExpertiseReversalEffectAndItsI.pdf); [The Guidance Fading Effect — Sweller (backward fading)](https://cogscisci.wordpress.com/wp-content/uploads/2019/08/sweller-guidance-fading.pdf)
- **The ICAP framework predicts learning rises from Passive → Active → Constructive → Interactive engagement, so generated pages should push learners up the engagement ladder (generate, not just receive).** _(confidence: high)_  
  Chi & Wylie (2014) categorize engagement by overt behavior: Passive (receiving), Active (manipulating/highlighting), Constructive (producing new ideas beyond the material, e.g. self-explaining, concept-mapping), Interactive (generative dialogue). Self-explanation also exposes knowledge gaps and supports metacognitive monitoring.  
  → *so what:* Add a Constructive (ICAP) element to pages: a 'why does this work?' / 'explain this step in your own words' prompt before revealing the answer — generated automatically from the concept. ### Recommendations
- Encode the principles as a two-layer eval rubric: (a) mechanical lint checks on each generated page (Mayer coherence/signaling/contiguity/redundancy, presence of a Bloom-verb objective, presence of >=1 retrieval item, ICAP level of the primary interaction, visual-does-cognitive-work check) and (b) an LLM-judge rubric for the harder-to-mechanize properties (objective↔activity↔assessment alignment, self-explanation quality, scaffolding fade). Make coherence, prerequisite-ordering, and 'has retrieval' hard gates.
- Make the ANALYSIS phase output an explicit prerequisite knowledge graph (concept nodes + prerequisite edges + a target Bloom level per node). Topologically sort it into the tier/category sitemap so SYNTHESIS can never place a concept before its prerequisites; reuse the same graph as a learner-facing concept map.
- Branch generation on the SETTINGS prior-knowledge slider because of the expertise-reversal effect: novices get pre-training term blocks, full + faded worked examples, and more scaffolding; advanced learners get problems-first with minimal guidance. A single 'one size' artifact should be penalized by evals.
- Require every bite-size page to climb the ICAP ladder to at least Constructive: include a predict-then-reveal step, a self-explanation prompt (Bisra g≈0.55, and prompt rather than provide), or a build-the-diagram interaction — not just click-to-reveal exposition.
- Design the OUTPUT-side metrics and workflow A/B tests around DELAYED retention and transfer (return-quizzes, spaced reviews), not just immediate engagement/time-on-page/satisfaction. Desirable-difficulties research shows immediate signals can be negatively correlated with durable learning; optimizing on them would select for fluent-but-shallow material.
- Treat the interactive visualization as load-bearing per dual-coding/multimedia: enforce that each visual encodes the concept's structure, place labels adjacent to the parts they describe (spatial contiguity), pace interactivity by the learner (segmenting), and forbid redundant on-screen text duplicating narration.
- Use Bloom verbs for constructive alignment: each page declares an objective with a Bloom verb and ships at least one assessment item at that cognitive level; the eval verifies the assessment actually targets the claimed level.

### Open questions
- How is 'prior knowledge' actually measured at intake? A self-reported slider is coarse, yet expertise reversal makes it the most consequential setting. Should the workflow include a short diagnostic pre-test (also a retrieval-practice benefit) to calibrate guidance level objectively?
- What is the unit of A/B testing and how is delayed retention captured without heavy learner commitment? Workflow-level A/B on durable-learning outcomes requires learners to return for a delayed test — what incentive/instrumentation makes that data collectable at scale?
- Which principles are mechanically checkable vs. require an LLM judge, and how do we validate the judge? Coherence/contiguity/retrieval-presence are lintable; scaffolding-fade, self-explanation quality, and objective-alignment likely need a judge that itself must be eval'd against human ratings.
- How aggressively to apply desirable difficulties given they reduce short-term satisfaction? There is a real tension between the founder's engagement metrics and durable-learning quality — what blend keeps learners motivated without sacrificing the difficulty that produces learning?
- Do these principles (mostly from text/video/classroom studies) transfer cleanly to interactive HTML/Canvas visualizations? The dual-coding and ICAP evidence is supportive but the specific medium (generative, interactive single-page viz) is under-studied; the product itself could become an eval testbed.
- How is the prerequisite knowledge graph validated for correctness? An LLM-inferred prerequisite ordering can be wrong; what cross-check (multi-criteria inference, expert spot-check, learner-failure signals) guards against bad sequencing?

### Verification verdicts

- [mostly-supported] Claim 2: Mayer's multimedia principles give ~15 directly checkable composition rules for any page that combines words and visuals.

## Multi-agent analysis -> synthesis architecture  
*overall trust: high*

For topic-synthesis, the strongest evidence-backed shape is a two-phase pipeline that cleanly separates an ANALYSIS sub-pipeline (decompose the topic, research with grounding, and structure results into a prerequisite/curriculum knowledge graph) from a SYNTHESIS sub-pipeline (compose one interactive page per graph node, then critique-and-refine). The orchestrator-worker pattern proven in Anthropic's multi-agent research system is the right backbone for the ANALYSIS phase: a lead/planner agent writes an explicit plan, spawns 3-5 parallel subagents with isolated context windows and detailed task briefs, and reconciles their condensed findings with a separate citation pass. This pattern beat a single-agent baseline by 90.2% on Anthropic's internal research eval but costs ~15x the tokens of chat, so multi-agent fan-out should be reserved for the breadth-first research stage, not every step. For SYNTHESIS, the most parallelizable shape is map-reduce over graph nodes (LangGraph's Send API spins up one worker per node dynamically), with each node generating an interactive artifact via an intermediate spec (plan -> code, in the spirit of ViviDoc's DocSpec) and passing through a reflection/critique-refine loop (Reflexion-style) gated by an LLM-as-judge rubric. Stanford's STORM is the closest published analogue to the whole product: it already does perspective-guided question-asking, retrieval-grounded research, outline/curriculum construction, then section-by-section writing with citations. The recommended concrete pipeline: Planner (decompose + curriculum graph) -> parallel Researcher subagents (agentic RAG, isolated context, condensed returns) -> Graph-builder (prerequisite DAG + per-node briefs) -> [for each node] map-reduce Synthesizer (spec -> interactive HTML) -> Critic (reflection + rubric judge) -> Citation/grounding pass -> Hub assembler (tiered sitemap). Use a strong model for planning/critique and cheaper models for execution to control the cost blow-up. Treat the curriculum graph as the durable intermediate artifact that makes the workflow inspectable, A/B-testable, and evaluable.

Anthropic's research evals use rubrics for factual accuracy, citation accuracy, completeness, source quality, and tool efficiency, started with a few test cases, and kept humans in the loop to catch biases (e.g. SEO-content-farm preference) automated metrics missed. For >10k examples, program-synthesized judges (PAJAMA) are recommended.  
  → *so what:* Build two rubric families. ANALYSIS evals score the curriculum graph and research: coverage/breadth, prerequisite-ordering correctness, factual + citation accuracy, source quality, absence of cycles. SYNTHESIS evals score each generated page: pedagogical scaffolding, interactivity correctness (does the visualization actually run/behave per spec), accuracy, and fit to the learner's settings (depth/prior-knowledge). Wire these judges in as the Critic's gate and as the offline metric for A/B testing entire workflows.  
  [LLMs-as-Judges: A Comprehensive Survey on LLM-based Evaluation Methods — arXiv 2412.05579](https://arxiv.org/html/2412.05579v2); [How we built our multi-agent research system — Anthropic](https://www.anthropic.com/engineering/multi-agent-research-system)

### Recommendations
- Adopt an explicit two-phase plan-and-execute pipeline with the curriculum graph as the contract between phases. ANALYSIS: Planner (decompose + perspective-guided question-asking, STORM-style) -> parallel agentic-RAG Researcher subagents (isolated context, 1-2k-token condensed returns) -> Graph-builder (prerequisite DAG with typed nodes + per-node briefs) -> citation/grounding verification. SYNTHESIS: map-reduce Send() one Synthesizer per node -> spec-then-code (ViviDoc DocSpec style) -> Critic (Reflexion-style reflection gated by an analytic rubric, capped iterations) -> Hub assembler (tiered sitemap with built/'soon' status).
- Use a tiered model strategy to contain the ~15x multi-agent token cost: strong model for Planner, Graph-builder, and Critic; cheaper models for Researcher execution and first-draft node synthesis; reserve multi-agent fan-out strictly for the breadth-first research stage.
- Make every intermediate artifact persistent and addressable (research bundle, curriculum graph, per-node spec, generated page, critique notes). This enables resumable execution, single-node regeneration, incremental 'soon->built' hub status, and is the substrate for both per-node engagement data and workflow-level A/B testing.
- Validate the curriculum graph as a hard gate before synthesis: no prerequisite cycles (topological sort succeeds), every node reachable, depth/breadth matches the user's settings, and each node has verified citations. A bad graph poisons every downstream page.
- Build two decomposed analytic LLM-as-judge rubric families (analysis-quality and synthesis-quality, including a check that interactive pages actually run and behave per their spec), seed them with a handful of human-reviewed cases, keep humans in the loop early to catch judge biases, and wire the same judges into both the Critic gate and offline workflow A/B evaluation.
- Treat the whole pipeline (planner prompt, researcher strategy, graph schema, synthesis spec, critic rubric) as a versioned 'workflow' object so the founder's requirement to A/B test entire workflows is satisfied by routing a topic+settings request to a workflow version and scoring its outputs with the shared eval rubrics.

### Open questions
- Granularity of fan-out in synthesis: is one subagent per leaf node optimal, or should categories/tiers be batched into a single worker to share context and reduce cost? Anthropic warns multi-agent loses on tightly-coupled tasks — how coupled are sibling concept pages?
- How much of the curriculum graph should be LLM-generated vs seeded from a retrieved canonical source (e.g. existing syllabi/textbook TOCs) given that research favors a hybrid expert+inference strategy for prerequisite edges? What is the cold-start quality on niche topics with thin web coverage?
- Should the Critic regenerate the interactive page, or only the DocSpec, when a page fails the rubric? Regenerating the spec is cheaper but may not fix code-level interactivity bugs that only surface at render/run time.
- What is the right ceiling on critique-refine iterations and research depth per node to keep within the user's stated time budget, and how should the Planner translate 'time budget' into subagent count and iteration caps?
- How is 'interactivity correctness' evaluated cheaply at scale — does each generated page need to be executed/rendered (headless browser) in the eval loop, and how does that cost interact with map-reduce throughput?
- Where does the human (the learner or an editor) enter the loop — Co-STORM-style mid-analysis to steer perspectives, or only post-synthesis feedback that feeds the founder's output-side data collection?

## Robust A/B testing of whole LLM workflows  
*overall trust: high*

A/B testing an entire LLM/agentic workflow (analysis -> synthesis -> rendered curriculum) is tractable if you treat the whole pipeline as a single, versioned, immutable artifact and make that version the experiment arm. The hard parts versus classic web A/B testing are: (1) outputs are stochastic and quality metrics are noisy and expensive to measure (LLM-judge or human), which inflates required sample sizes; (2) the randomization unit must be the learner/session, not the request, because one learner sees one whole generated curriculum; and (3) quality signals arrive slowly and sparsely, so you need variance reduction and disciplined sequential analysis to read results without peeking yourself into false positives. The good news: the experimentation tooling needed already exists off the shelf. Statsig, GrowthBook, and Eppo all implement CUPED variance reduction and sequential/always-valid inference; Statsig reports CUPED-driven variance cuts on the order of ~50% (doubling effective traffic) and offers mSPRT sequential testing at ~84% relative power. For the offline/eval side, Braintrust gives immutable, comparable experiment snapshots with LLM-as-judge scoring and CI regression gating. The recommended architecture is a two-layer loop: an offline eval harness (Braintrust) that scores whole-workflow versions on a fixed dataset to gate which arms even reach production, then an online experiment platform (Statsig or Eppo, warehouse-native) that randomizes learners across workflow versions, uses CUPED on a pre-period engagement covariate, defines cost/latency/refusal guardrails, and reads results under a sequential framework. Where the primary output is a ranking-like artifact (e.g. which topics/visualizations to surface), interleaving can deliver 10-100x sensitivity over A/B for that sub-decision, per Netflix and Airbnb. Be honest about limits: published, peer-reviewed methodology specifically for A/B testing *whole agentic workflows* (not single prompts) is still thin; most of the rigor is borrowed from web experimentation and LLM-eval statistics and must be adapted.

### Key findings

- **The randomization unit must be the learner/session and the experiment arm must be a whole pinned workflow version, not an individual prompt or model swap.** _(confidence: high)_  
  Industry guidance on versioning agentic systems notes that 'version 1.2 of the agent' is ambiguous: it could mean the prompt, tool interface, model weights, context dataset, or memory schema changed, and that a feature flag which 'redirects an agent to a new tool may ripple into entirely new output structures.' Because a single learner consumes one entire generated curriculum end-to-end, randomizing at the request level would mix arms within one user's experience and break attribution. The clean pattern is to bundle the full pipeline (model ids, every prompt, retrieval config, tool set, orchestration graph, render templates) into one immutable, content-addressed version and make that hash the arm; Statsig's own LLM guidance assigns users so 'each group experiences only one variant.' Braintrust formalizes the offline analog: 'promote it to an experiment to capture an immutable snapshot... the immutable, comparable record of your eval runs.'  
  → *so what:* topic-synthesis should define a single workflow-version manifest (a config object hashing model ids + all prompts + retrieval/tool config + orchestration DAG + transitions-dev render templates). The A/B platform flag should return that version id, the orchestrator should pin every agent call to it, and randomization should be by learner id (sticky) so each learner sees exactly one workflow version for the whole curriculum.  
  [Why versioning AI agents is the CIO's next big challenge | CIO](https://www.cio.com/article/4056453/why-versioning-ai-agents-is-the-cios-next-big-challenge.html); [Beyond prompts: A data-driven approach to LLM optimization (Statsig)](https://www.statsig.com/blog/llm-optimization-online-experimentation); [Evaluate systematically - Braintrust](https://www.braintrust.dev/docs/evaluate)
- **LLM workflow outputs are stochastic even at temperature 0, so the experiment design must absorb output noise rather than assume reproducibility.** _(confidence: high)_  
  Practitioner writeups note that 'even at temperature zero, LLMs produce slightly different outputs due to floating-point precision and MoE routing, with small behavior changes in early steps cascading through subsequent steps' in multi-step agents. Statsig's LLM guidance concedes only that 'because LLM outputs can be stochastic, teams should ensure adequate sample sizes and carefully controlled conditions.' This stochasticity is multiplicative across an agentic DAG: a minor variation in the analysis agent's topic decomposition changes everything synthesis produces downstream.  
  → *so what:* Do not try to make the workflow deterministic for testing; instead (a) treat each learner's run as one noisy draw and rely on across-learner aggregation, (b) log the full trace + the version hash + sampled seed so any single curriculum can be reproduced for debugging, and (c) expect higher variance than a typical web A/B test, which directly drives the need for variance reduction and larger samples (next findings).  
  [A/B Testing AI Agents | Experiment with Agent Behavior](https://www.buildmvpfast.com/blog/ab-testing-ai-agents-experiment-production-behavior-2026); [Beyond prompts: A data-driven approach to LLM optimization (Statsig)](https://www.statsig.com/blog/llm-optimization-online-experimentation)
- **CUPED variance reduction is the single highest-leverage technique for noisy LLM quality metrics, and it is built into the major platforms by default.** _(confidence: high)_  
  CUPED (Controlled-experiment Using Pre-Experiment Data), introduced by Microsoft in 2013, adjusts each user's outcome using their pre-experiment behavior via regression; the variance of the metric Y is reduced by a factor of (1 - Corr(X,Y)^2), so a strong pre-period covariate yields large gains. Microsoft/Bing reported ~50% variance reduction, 'equivalent to doubling traffic or halving the experiment duration.' Statsig applies CUPED to all Scorecard metrics by default: 'CUPED can significantly shrink confidence intervals and p-values, ultimately reducing the sample size and duration required.' CUPED is mathematically a repeated-measures / matched-pairs-style design, removing pre-existing between-user differences.  
  → *so what:* Capture a pre-experiment engagement covariate for every learner before they enter an experiment (e.g. prior session count, prior completion rate, prior topics finished, or a baseline quiz score) so CUPED has a high-correlation X. This is the cheapest way to roughly halve the sample size needed to detect a workflow improvement, which matters enormously when each observation is an expensive multi-agent generation. Pick Statsig/Eppo/GrowthBook partly because CUPED is native.  
  [Improving the Sensitivity of Online Controlled Experiments (CUPED, Microsoft 2013)](https://exp-platform.com/Documents/2013-02-CUPED-ImprovingSensitivityOfControlledExperiments.pdf); [Matched pairs design in statistics: explained | GrowthBook](https://www.growthbook.io/insights/matched-pairs-design-statistics-explained); [Sequential Testing on Statsig (notes CUPED applied to all Scorecard metrics)](https://www.statsig.com/blog/sequential-testing-on-statsig)
- **Peeking at running experiments inflates false positives unless you use a sequential/always-valid framework; Statsig's mSPRT trades ~16% power for the right to stop early safely.** _(confidence: high)_  
  Repeated significance testing inflates type-I error: Statsig's die analogy is that a 5% daily false-positive chance compounds 'to 30%' over a week of peeking. Sequential methods fix this. Statsig uses mSPRT (mixture Sequential Probability Ratio Test) and reports it delivers 84% statistical power relative to fixed-horizon testing, 58% probability of stopping by half the planned duration, and an actual false-positive rate of 0.4-1.1% (below 5%); the documented downside is 'lower precision' and a tendency toward 'systematically overestimating' effect sizes when you stop early on a positive. The academic foundations are always-valid inference (Johari et al.) and e-values/e-processes, which 'remain valid even when decisions to perform later experiments depend on earlier data.'  
  → *so what:* Pre-register a fixed primary readout horizon for the headline quality decision, but enable a sequential framework (mSPRT or always-valid CIs) on guardrail metrics so you can halt a workflow version that is actively harming learners or burning cost without waiting. Discount early-stopped positive effect sizes when forecasting impact. Do NOT let the team eyeball the dashboard daily and ship on the first green - that is exactly the peeking that sequential testing exists to neutralize.  
  [Sequential Testing on Statsig](https://www.statsig.com/blog/sequential-testing-on-statsig); [Always Valid Inference: Bringing Sequential Analysis to A/B Testing (Johari et al.)](https://arxiv.org/pdf/1512.04922); [E-values - Wikipedia](https://en.wikipedia.org/wiki/E-values)
- **For noisy/expensive quality metrics, plan for hundreds of evaluation units and explicitly model LLM-judge error, because the judge's noise widens the effective confidence interval.** _(confidence: high)_  
  Recent LLM-eval research finds exact accuracy 'begins to stabilize between 250 and 500 samples,' with standard error dropping sharply past 250, suggesting 250+ samples per arm as a floor for reliable offline eval. For online quality measured by an imperfect LLM judge, the 'Noisy but Valid' line of work shows you can recover valid inference by using a small human-labeled calibration set to estimate the judge's true-positive and false-positive rates, then deriving variance-corrected thresholds that 'theoretically guarantee finite-sample Type-I error control despite calibration uncertainty.' Separate work decomposes eval noise into prediction noise (same input, different output), data/sampling noise, and combined total noise.  
  → *so what:* For topic-synthesis: (1) treat 250-500 graded curricula per arm as a rough offline-eval floor before trusting a score; (2) keep a human-graded calibration set (say 100-200 items) to estimate your LLM judge's TPR/FPR and correct online quality estimates rather than treating judge labels as ground truth; (3) budget sample-size/power assuming quality metrics are noisier than clicks - the judge adds variance on top of output stochasticity. Online behavioral proxies (completion, time-on-page, quiz pass rate) are cheaper and less noisy than judge scores and should carry the primary readout when available.  
  [Noisy but Valid: Robust Statistical Evaluation of LLMs with Imperfect Judges](https://arxiv.org/abs/2601.20913); [Applying Statistics to LLM Evaluations - Cameron Wolfe](https://cameronrwolfe.substack.com/p/stats-llm-evals); [Measuring all the noises of LLM Evals](https://arxiv.org/html/2512.21326v2)
- **Every workflow experiment needs guardrail metrics (cost, latency, refusal/error rate) alongside the primary quality metric, because LLM workflow 'wins' frequently trade quality for large cost or latency regressions.** _(confidence: high)_  
  Statsig's LLM guidance recommends pairing 'a primary metric (like user retention) with guardrail metrics (like latency)' and warns 'a 3% increase in user satisfaction may not justify a 50% spike in cost.' Guardrail research confirms added LLM processing can impose meaningful latency and cost overhead, with practical alerting like p95>200ms warning / p99>500ms critical, and that inference is '80-90% of total compute spend over a model's production lifecycle.' Static-threshold alerting is fragile (low thresholds over-fire, high thresholds miss regressions), so guardrails should be statistical, not just fixed cutoffs.  
  → *so what:* topic-synthesis must instrument per-curriculum cost (tokens x price across all agents), wall-clock generation latency, and a failure/refusal/empty-render rate, and register them as guardrails on every workflow experiment. A new analysis+synthesis pipeline that adds 4 more agent calls might marginally raise quality while doubling cost and pushing generation past a tolerable wait - the guardrails are what stop you shipping it. Wire guardrail breaches to the sequential framework so a clearly harmful arm can be auto-stopped.  
  [Beyond prompts: A data-driven approach to LLM optimization (Statsig)](https://www.statsig.com/blog/llm-optimization-online-experimentation); [No Free Lunch with Guardrails](https://arxiv.org/pdf/2504.00441); [LLM Monitoring Best Practices: Complete Guide for 2026](https://openobserve.ai/blog/llm-monitoring-best-practices/)
- **Where the decision reduces to which items/visualizations to surface (a ranking sub-problem), interleaving offers 10-100x more sensitivity than A/B and should pre-screen workflow variants before full A/B.** _(confidence: medium)_  
  Netflix, Airbnb, and Thumbtack use interleaving for ranking experiments. Reported gains are dramatic: one experiment needed '~400 samples to reach a 90% agreement rate while the A/B test requires ~40,000' - a 100x sample-efficiency improvement; interleaving 'increased the sensitivity of experiments by a factor of up to 100.' It works by merging both variants' outputs into one list shown to the same user and attributing actions back to each source, removing between-user variance entirely. The known limitation: interleaving measures *preference between rankings*, not downstream business outcomes, so it is used to cheaply narrow many candidates before a confirmatory A/B test on the winner.  
  → *so what:* If topic-synthesis surfaces a ranked curriculum hub (tiers -> categories -> bite-size pages) or competing visualization candidates per concept, you can interleave two workflow variants' page/topic orderings within one learner's hub and measure which variant's items get engaged with - screening many candidate workflows fast - then run a full learner-randomized A/B on the finalist to confirm real learning-outcome lift. Don't use interleaving as the final arbiter of learning outcomes; use it as a cheap front filter.  
  [Innovating Faster on Personalization Algorithms at Netflix Using Interleaving](https://netflixtechblog.com/interleaving-in-online-experiments-at-netflix-a04ee392ec55); [Beyond A/B Test: Speeding up Airbnb Search Ranking Experimentation through Interleaving](https://airbnb.tech/data/beyond-a-b-test-speeding-up-airbnb-search-ranking-experimentation-through-interleaving/); [Sensitive and Scalable Online Evaluation with Theoretical Guarantees](https://arxiv.org/pdf/1711.09454)
- **Run a two-layer loop: offline whole-workflow evals (Braintrust) gate which versions reach production, then a warehouse-native online platform (Statsig/Eppo/GrowthBook) randomizes learners across versions - the platforms differ enough that selection matters.** _(confidence: medium)_  
  Braintrust scopes the offline layer: online scoring 'evaluates production traces automatically as they're logged, running asynchronously with no impact on latency,' experiments are 'immutable, comparable,' and CI gating blocks a PR when 'scores fall below defined thresholds.' For the online layer, 2025/2026 comparisons characterize: GrowthBook as the open-source/self-hostable, warehouse-native flexible leader; Statsig as all-in-one with generous free tier (free to 10M events/mo) and native CUPED + sequential testing; Eppo as warehouse-native with strong statistical rigor but weak feature-flagging; LaunchDarkly as feature-flag-first with shallower experimentation and high per-context pricing ($60k-$240k/yr quoted at mid-market). Eppo and GrowthBook being warehouse-native matters because your learner-outcome metrics will live in your own analytics DB.  
  → *so what:* Concretely: use Braintrust (or equivalent) as the offline gate that scores whole-workflow versions on a fixed topic dataset and blocks regressions pre-merge; use a warehouse-native online platform (Statsig if you want CUPED+sequential+flags out of the box and a cheap start; Eppo/GrowthBook if you want results computed over your own warehouse metrics) to randomize learners across workflow versions. Avoid LaunchDarkly as the experimentation engine - it is a flag system, not a rigorous experiment analyzer, though it could supply the version-serving flag.  
  [Evaluate systematically - Braintrust](https://www.braintrust.dev/docs/evaluate); [The best A/B testing platforms of 2025 | GrowthBook](https://www.growthbook.io/blog/the-best-a-b-testing-platforms-of-2025); [Top 7 alternatives to Eppo for Experimentation | Statsig](https://www.statsig.com/comparison/alternatives-to-eppo-experimentation)
- **Peer-reviewed, workflow-specific A/B methodology is still thin; most rigor is adapted from web experimentation and LLM-eval statistics, and the emerging 'LLM agents as test traffic' idea is promising but unproven for outcome decisions.** _(confidence: medium)_  
  The literature that exists specifically on testing agentic workflows is mostly practitioner blogs plus a small set of papers. AgentA/B (arXiv 2504.09723) assigns '1,000 LLM agents to treatment and control variants' on a live site and shows agents 'detect subtle design differences and generate measurably different outcomes' - useful for cheap synthetic pre-tests but not a substitute for real-learner causal inference. A surrogacy framework for LLM-based A/B testing (arXiv 2606.17165) is recent and unvalidated at scale. By contrast the variance-reduction, sequential-testing, and judge-noise results are mature. So the honest position is that whole-workflow A/B testing is a *composition* of well-established pieces, not a turnkey, peer-blessed method.  
  → *so what:* Design topic-synthesis's experimentation as an assembly of proven components (learner-level randomization + CUPED + sequential analysis + guardrails + judge calibration) rather than waiting for a canonical 'agentic A/B' standard. Optionally use LLM-agent synthetic traffic (AgentA/B-style) as a fast, low-cost smoke test to discard obviously-worse workflow variants before spending real learner traffic - but require a real human-learner A/B for any ship decision on learning outcomes.  
  [AgentA/B: Automated and Scalable Web A/B Testing with Interactive LLM Agents](https://arxiv.org/html/2504.09723v3); [Statistical Foundations of LLM-based A/B Testing: A Surrogacy Framework](https://arxiv.org/html/2606.17165); [A/B Testing Strategies for AI Agents (Maxim)](https://www.getmaxim.ai/articles/a-b-testing-strategies-for-ai-agents-how-to-optimize-performance-and-quality/)

### Recommendations
- Make the experiment arm a content-addressed workflow-version manifest: hash {model ids, every agent prompt, retrieval/tool config, orchestration DAG, transitions-dev render templates} into one id. The A/B flag returns that id; the orchestrator pins all agent calls to it. This is what makes 'a version = the experiment arm' literally true and reproducible.
- Randomize at the learner/session level with sticky assignment so each learner experiences exactly one whole workflow version across their entire generated curriculum. Never randomize per-request inside a single learner's curriculum.
- Adopt the two-layer loop: (1) offline Braintrust evals on a fixed topic dataset to gate/regression-block workflow versions pre-merge using LLM-judge + human-calibrated scores; (2) online learner-randomized experiment on a warehouse-native platform for real outcome lift. Start with Statsig (native CUPED + mSPRT + flags + free tier) unless you specifically need warehouse-native compute over your own metrics, in which case evaluate Eppo or self-hosted GrowthBook.
- Capture a high-correlation pre-experiment covariate per learner (prior completion rate, prior topics finished, baseline quiz score) and rely on the platform's CUPED to roughly halve required sample size - critical because each observation is an expensive multi-agent generation.
- Define the primary metric as a cheap, low-noise behavioral outcome (curriculum completion, quiz pass rate, return/retention) and relegate expensive LLM-judge quality scores to a secondary/offline role; calibrate the judge against a 100-200 item human-labeled set and apply variance-corrected thresholds rather than trusting raw judge labels.
- Register cost-per-curriculum (summed tokens x price across all agents), end-to-end generation latency, and failure/refusal/empty-render rate as guardrail metrics on every experiment, and wire them to the sequential framework so a clearly harmful arm auto-stops.
- Pre-register a fixed primary-readout horizon and forbid shipping on the first green dashboard; use mSPRT/always-valid CIs only for early-stop safety on guardrails, and discount early-stopped positive effect sizes (they over-estimate).
- If you expose a ranked curriculum hub or competing per-concept visualizations, use interleaving to cheaply pre-screen many workflow variants (10-100x sensitivity), then confirm the finalist with a full learner-randomized A/B on actual learning outcomes.
- Use AgentA/B-style LLM synthetic traffic only as a pre-flight smoke test to kill obviously-worse variants cheaply; require a real human-learner experiment for any learning-outcome ship decision.
- Log full per-run traces tagged with the workflow-version hash and sampled seeds so any single generated curriculum can be reconstructed for debugging and for judge calibration.

### Open questions
- What is the primary outcome metric for a learning product, and how long is its measurement window? Completion is fast but shallow; durable learning (retention quizzes, spaced re-tests) is the real goal but arrives days/weeks later, which drastically changes power and experiment duration.
- How many active learners will topic-synthesis realistically have early on? If traffic is low, full online A/B may be underpowered for weeks - pushing weight onto offline evals, interleaving pre-screens, and CUPED. Need an honest power calculation against expected effect sizes.
- Should the experiment arm pin the underlying model version too? Foundation models update underneath you; if the model drifts mid-experiment, the 'version' is no longer immutable. Decide whether to pin to dated model snapshots.
- What is the unit for quality - the whole curriculum, each tier, or each individual concept page? This determines whether you analyze at curriculum level (few noisy units) or page level (many correlated units needing clustered/ratio-metric variance handling).
- Is there a meaningful interference/SUTVA risk (e.g. shared topic caches, learners discussing curricula, or one generated hub influencing another)? Cross-arm contamination would bias estimates and may require cluster randomization.
- Which LLM judge, and how often is its calibration set refreshed? Judge drift over time silently corrupts longitudinal quality comparisons across experiments.
- Do we need the same statistical engine for offline (Braintrust) and online (Statsig/Eppo) layers, or is it acceptable that they use different inference methods, risking offline-vs-online disagreement on what 'better' means?

## Evals for analysis & synthesis (incl. learning-material quality)  
*overall trust: high*

For topic-synthesis, the eval problem splits cleanly: (1) ANALYSIS quality (the intermediate research/decomposition step), which is closer to a RAG/groundedness problem and has good reference-free tooling, and (2) SYNTHESIS / learning-material quality, which is a near-open problem with no ground-truth label and must lean on rubric-based LLM-as-judge plus human-in-the-loop calibration and, eventually, real learner-outcome signal. The field has converged on a repeatable recipe: define crisp, mostly-binary rubric criteria with a single domain-expert "benevolent dictator", hand-label a 200-500 example gold set, align an LLM judge to those labels until Cohen's kappa / agreement clears ~0.6-0.8 (using TPR/TNR not raw accuracy), and then mitigate the well-documented judge biases (position, verbosity/length, self-preference) via order randomization, score-then-rank, ensembles, and explicit length penalties. Groundedness/faithfulness for the analysis stage is the most tractable, label-free part and should be evaluated against the retrieved sources (Ragas-style faithfulness, MiniCheck/AlignScore-class checkers). For pedagogical quality, no single metric suffices: combine an LLM-judged rubric (accuracy, coverage, scaffolding, clarity, engagement) with cheap deterministic guards (readability bands, prerequisite-ordering checks, link/code validity) and treat readability formulas as guardrails, not quality scores. Crucially, the founder's two adjacent asks — workflow A/B testing and output-side engagement data — are the only path to a true quality signal: offline rubric evals gate regressions and rank workflow variants cheaply, but the gold standard is online experiments where whole workflow variants compete on learner outcomes (completion, return, self-rated understanding), with Bradley-Terry/Elo aggregation of pairwise preferences as the ranking layer. Recommended stack: Braintrust as the platform of record (traces, datasets, scorers, experiments, generous free tier) with promptfoo or DeepEval/Ragas as the CI-gating layer; build the gold set and the rubric first, before the generator is even good.

### Key findings

- **Reference-free LLM-as-judge (e.g. G-Eval) is the only practical way to score analysis/synthesis quality with no ground-truth label, but it is a measuring instrument that must itself be calibrated before its scores mean anything.** _(confidence: high)_  
  Classic reference-based metrics (BLEU/ROUGE) correlate poorly with human judgment on creative/synthesis tasks, which is why the field shifted to LLM judges. G-Eval (Liu et al., 2023) uses chain-of-thought to expand a natural-language criterion into explicit evaluation steps, then form-fills a score, optionally weighting by token log-probabilities; it was the first to show strong human alignment on summarization and dialogue. But practitioners are emphatic: 'Calibration is the step where you measure whether the judge's score correlates with what humans actually think. Without calibration, the score is decorative.'  
  → *so what:* topic-synthesis must treat the judge as a versioned artifact with its own eval: build it, align it to human labels, and re-check it monthly. Do NOT ship a 1-5 LLM rubric score as a quality KPI until it has demonstrated agreement with a human gold set. Budget eng time for the judge, not just the generator.  
  [G-Eval: NLG Evaluation using GPT-4 with Better Human Alignment (arXiv 2303.16634)](https://arxiv.org/abs/2303.16634); [LLM-as-Judge Best Practices in 2026: Calibration, Bias, and Cost (Future AGI)](https://futureagi.com/blog/llm-as-judge-best-practices-2026); [G-Eval: The Definitive Guide (Confident AI / DeepEval)](https://www.confident-ai.com/blog/g-eval-the-definitive-guide)
- **LLM judges carry systematic biases — position, verbosity/length, and self-preference — large enough to flip verdicts, so the eval harness needs explicit de-biasing, not just a good prompt.** _(confidence: high)_  
  In pairwise judging, simply swapping presentation order can shift accuracy by >10%; judge-model choice is the dominant factor in positional bias. Judges also prefer longer/more fluent answers irrespective of substance (verbosity bias) and prefer outputs from their own model family (self-preference). Documented mitigations: randomize slot order and aggregate; score each output independently against the rubric and derive the pairwise verdict post-hoc; use multi-judge ensembles (which disagree on position); and explicitly instruct 'penalize unnecessary length; favor concise correct answers' plus length-normalize density metrics.  
  → *so what:* Because A/B-testing whole workflows will lean heavily on pairwise judge comparisons, bake de-biasing into the harness from day one: always run A-vs-B and B-vs-A, prefer score-then-rank over raw pairwise, and use a judge from a different model family than the generator to blunt self-preference. Otherwise the workflow leaderboard will be measuring verbosity and order, not learning quality.  
  [Justice or Prejudice? Quantifying Biases in LLM-as-a-Judge (arXiv 2410.02736)](https://arxiv.org/html/2410.02736v1); [Self-Preference Bias in LLM-as-a-Judge (arXiv 2410.21819)](https://arxiv.org/pdf/2410.21819); [LLM-as-Judge Best Practices in 2026 (Future AGI)](https://futureagi.com/blog/llm-as-judge-best-practices-2026)
- **The calibration workflow is concrete and well-specified: a 200-500 example human-labeled gold set, 2-3 labelers, and Cohen's kappa (Krippendorff's alpha for 3+) with actionable thresholds — and you should prefer binary pass/fail over Likert scales.** _(confidence: high)_  
  Recommended recipe: sample 100-300 (some say 200-500) diverse production traces, have 2-3 humans label them against the rubric, then compute Cohen's kappa between judge and majority human label. Thresholds: inter-annotator kappa <0.4 = ambiguous rubric, rewrite it; 0.4-0.6 weak but tunable; >0.6 acceptable; >0.8 strong. Recalibrate monthly and refresh the oldest ~100-200 entries quarterly. Hamel Husain argues against 1-5 scales entirely ('if your evals score on a 1-5 scale, you're doing it wrong') because numeric ratings aren't actionable; instead use binary pass/fail with a written critique, a single domain-expert 'benevolent dictator' to set standards, error analysis to find root-cause failure clusters, and track judge alignment via precision/recall (TPR/TNR), not raw accuracy, because the label distribution is imbalanced.  
  → *so what:* For topic-synthesis: decompose 'analysis quality' and 'synthesis quality' into a handful of binary checks (each pass/fail with a critique) rather than one fuzzy 1-10 score. Recruit/appoint one pedagogy expert as the standard-setter. The first deliverable of the eval workstream is the labeled gold set, built before optimizing the generator — kappa is the gate that tells you whether the rubric is even well-defined.  
  [Creating a LLM-as-a-Judge That Drives Business Results — Hamel Husain](https://hamel.dev/blog/posts/llm-judge/); [LLM-as-Judge Best Practices in 2026 (Future AGI)](https://futureagi.com/blog/llm-as-judge-best-practices-2026); [LLM-as-a-Judge vs Human Evaluation (Galileo)](https://galileo.ai/blog/llm-as-a-judge-vs-human-evaluation)
- **The ANALYSIS stage is the most tractable part to evaluate label-free: treat it as a groundedness/faithfulness problem against the retrieved sources, using the four-metric taxonomy (groundedness, faithfulness, factuality, consistency).** _(confidence: high)_  
  Braintrust's hallucination-eval taxonomy maps cleanly: groundedness validates claims against retrieved documents, faithfulness checks output vs source text, factuality checks against world knowledge, and consistency measures agreement across repeated generations (semantic entropy / self-consistency) for open-ended cases with no reference. Reference sources vary by task: RAG QA uses retrieved context, summarization uses the source doc, free-form uses consistency. Specialized faithfulness checkers exist (Ragas faithfulness/context-precision/recall; MiniCheck, AlignScore, SUMMAC, TRUE, FaithBench, RAGTruth from the summarization-factuality literature; fine-tuned detectors like Patronus Lynx / Galileo Luna for speed).  
  → *so what:* For the multi-agent research/analysis step, log every claim with its source and run a groundedness scorer (Ragas-style or a MiniCheck-class NLI checker) over each claim-source pair — this catches the most damaging failure (confidently teaching something false) without needing any gold label. Make 'every factual claim is supported by a cited source' a hard gate before synthesis even runs.  
  [AI hallucination evaluations: metrics and methods 2026 (Braintrust)](https://www.braintrust.dev/articles/ai-hallucination-evaluations-metrics-methods-2026); [Evaluation & Hallucination Detection for Abstractive Summaries — Eugene Yan](https://eugeneyan.com/writing/abstractive/); [awesome-hallucination-detection (EdinburghNLP)](https://github.com/EdinburghNLP/awesome-hallucination-detection)
- **There is real, recent research on rubrics for AI-generated educational/pedagogical quality, and it converges on a stable set of dimensions you can adopt directly — but findings warn pedagogical quality is multi-dimensional and AI is weakest on metacognition.** _(confidence: medium)_  
  A multi-LLM agent workflow for procedural learning (arXiv 2507.05528) assesses tutoring with the dimensions Clarity, Truthfulness, Engagement, Coherence, Depth, Relevance, Progress, and Naturalness. A large study of AI vs human formative feedback used a unified rubric of accuracy (faithful to source), coverage (thoroughness over key concepts), and learning-science principles, and found AI feedback comparable to human but both weak on metacognitive support. MDPI work on AI-generated educational resources built an index spanning content, expression, and user/technical quality via Delphi + AHP. An AAAI HCOMP study compared crowdsourced vs expert vs AI rubric application for educational quality.  
  → *so what:* Adopt a synthesis rubric grounded in this literature rather than inventing one: accuracy, coverage/completeness, scaffolding (prerequisite ordering & progressive difficulty), clarity, engagement, and appropriateness-to-settings (depth/prior-knowledge/format/time-budget the user selected). Explicitly add a metacognition/active-recall check since that's the documented AI weak spot — and since the founder's artifact is interactive visualizations, add a check that the visualization actually illustrates the concept (judge the page, not just the text).  
  [Conversational Education at Scale: A Multi-LLM Agent Workflow for Procedural Learning and Pedagogic Quality Assessment (arXiv 2507.05528)](https://arxiv.org/pdf/2507.05528); [Can students judge like experts? Pedagogical quality of AI and human personalized formative feedback (ScienceDirect)](https://www.sciencedirect.com/science/article/pii/S2666920X25001730); [Evaluating the Quality of AI-Generated Digital Educational Resources (MDPI Systems 13(3):174)](https://www.mdpi.com/2079-8954/13/3/174)
- **Readability formulas (Flesch-Kincaid etc.) are cheap deterministic guardrails, not quality metrics — they ignore vocabulary diversity, subject complexity, and coherence, and disagree wildly across formulas.** _(confidence: high)_  
  FKGL scores only sentence length and syllable count; documented limitations include overlooking vocabulary diversity, tone, and subject complexity, and large cross-formula discrepancies (one study saw FKGL range 2.2 to 36 on comparable material). The defensible use is matching text to a target grade band, not ranking quality. Continuous-eval and similar harnesses expose FK readability as a deterministic metric precisely because it's fast and reproducible.  
  → *so what:* Use readability as a guardrail tied to the user's selected prior-knowledge/depth setting (e.g. 'beginner' material must land in a target FK band) and as a regression tripwire, but never let it stand in for the LLM-judged pedagogical rubric. Pair it with other cheap deterministic checks: code/link validity, presence of an interactive viz, prerequisite-before-dependent ordering in the sitemap.  
  [Flesch–Kincaid Readability metric (Continuous Eval / Relari docs)](https://continuous-eval.docs.relari.ai/metrics/generation/deterministic/flesch_kincaid_readability/); [Evaluating Text Complexity and Flesch-Kincaid Grade Level (ResearchGate)](https://www.researchgate.net/publication/322020433_Evaluating_Text_Complexity_and_Flesch-Kincaid_Grade_Level); [Flesch Reading Ease and the Flesch Kincaid Grade Level (Readable)](https://readable.com/readability/flesch-reading-ease-flesch-kincaid-grade-level/)
- **Pairwise comparison with Bradley-Terry / Elo aggregation is the right ranking layer for comparing whole-workflow variants, and it's more reliable than absolute scoring for the no-ground-truth synthesis problem.** _(confidence: high)_

Newer work (Arena-Lite, Inclusion Arena, Tournament of Prompts using Elo) shows efficient pairwise tournaments rank variants reliably with far fewer comparisons. Pairwise sidesteps the hard problem of an absolute 'is this good curriculum?' score by only asking 'is A better than B?', which both humans and judges do more reliably.  
  → *so what:* For the founder's 'A/B test entire workflows' requirement: don't compare workflows on absolute rubric averages — run pairwise judge (and human) battles between workflow-variant outputs on the same topic, aggregate with Bradley-Terry to get a ranked leaderboard with confidence intervals, and use active sampling so you spend comparisons on the close calls. Combine an LLM-judge battle (cheap, offline) with a learner-preference battle (expensive, online).  
  [Chatbot Arena: An Open Platform for Evaluating LLMs by Human Preference (arXiv 2403.04132)](https://arxiv.org/pdf/2403.04132); [Bradley-Terry Model: Converting Preferences to Rankings (Brenndoerfer)](https://mbrenndoerfer.com/writing/bradley-terry-model-pairwise-preferences-rankings); [Arena-Lite: Efficient and Reliable LLM Evaluation via pairwise comparison (EMNLP 2025)](https://aclanthology.org/2025.emnlp-main.360.pdf)

### Recommendations
- Build the gold set FIRST, before optimizing the generator. Hand-curate ~50-100 diverse (topic x settings) inputs, run the current workflow, and have one pedagogy expert plus 1-2 others label outputs against the rubric with written critiques. This dataset is the foundation for judge calibration, CI gating, and workflow A/B.
- Split evals by stage. ANALYSIS: a label-free groundedness gate — every claim must cite a source; score claim-source support with a Ragas/MiniCheck-class faithfulness checker; hard-fail synthesis if unsupported claims exist. SYNTHESIS: an LLM-judged pedagogical rubric plus deterministic guardrails.
- Adopt a concrete STARTER RUBRIC. Analysis quality (each binary pass/fail + critique): (1) Groundedness — every claim supported by a cited source; (2) Coverage — all core subtopics/prerequisites of the topic are identified, none critical missing; (3) Correctness — no factual errors vs world knowledge; (4) Structure — decomposition is a valid prerequisite DAG (no concept depends on a later one); (5) Settings-fit — depth/scope matches the user's prior-knowledge & time-budget. Synthesis quality: (1) Accuracy — faithful to the analysis, no new hallucinations; (2) Completeness/Coverage — each tier/category page covers its concept; (3) Scaffolding — pages ordered by increasing difficulty, prerequisites before dependents; (4) Clarity — plain, well-structured explanation; (5) Readability-band — Flesch-Kincaid lands in the band implied by the prior-knowledge setting (deterministic); (6) Engagement/Active-recall — includes an interactive element / check-for-understanding (the documented AI weak spot is metacognition, so test for it); (7) Visualization-fidelity — the interactive viz actually illustrates the concept and renders without errors (judge the rendered page, plus a deterministic JS/render check).
- Prefer binary criteria with critiques over 1-5 Likert scores; aggregate to a per-artifact pass-rate. This is more actionable, less gameable, and aligns with the practitioner consensus.
- Calibrate the judge as a first-class artifact: align it to the human gold labels until Cohen's kappa/agreement clears ~0.6 (target >0.8), measure with TPR/TNR not accuracy, version the judge prompt, recalibrate monthly, and refresh the gold set quarterly. Treat kappa <0.4 as a signal the rubric criterion is ambiguous and must be rewritten.
- De-bias every judge call: run pairwise comparisons in both orders and aggregate (or score-then-rank), explicitly instruct the judge to penalize verbosity and length, and use a judge model from a DIFFERENT family than the generator to blunt self-preference.
- For workflow A/B testing, compare variants pairwise (not on absolute averages): LLM-judge battles offline for cheap pre-ranking, learner-preference battles online; aggregate both with Bradley-Terry to get a ranked leaderboard with confidence intervals, using active sampling on close calls.
- Design eval and output-telemetry together. Stamp every learner session with a workflow-variant id; log proxy outcomes (page completion, dwell, return, self-rated understanding, active-recall/quiz performance); canary new workflows with a guardrail that auto-aborts if an online metric breaches a threshold. Offline rubric = leading indicator; learner outcomes = verdict.
- Tooling: Braintrust as platform of record (traces, datasets, scorers, experiments, human annotation; bt CLI skill available) + promptfoo or DeepEval in CI to gate PRs against the gold set + Ragas-style faithfulness for the analysis stage; evaluate Inspect AI for structured rubric eval plans. Don't build eval infra from scratch.
- Add cheap deterministic guards that catch high-impact failures with no LLM cost: prerequisite-ordering check over the sitemap DAG, broken-link/code-validity check, visualization renders-without-error check, and readability-band check keyed to the settings.

### Open questions
- What is the single proxy 'north-star' learning-outcome metric? Completion and dwell are gameable; self-rated understanding is biased; a real comprehension check (auto-generated quiz / active recall) is more valid but adds product surface and its own eval problem. Decide before instrumenting.
- Who is the 'benevolent dictator' pedagogy expert that sets the rubric standard, and how do we secure enough of their labeling time for the initial gold set and monthly recalibration?
- How do we get ground truth on 'is this curriculum correct and complete' for arbitrary user topics, where no in-house expert exists? Options: cross-model consensus, retrieval-grounded fact-checking, or flagging low-confidence topics for human review — each has failure modes.
- How is the gold set stratified across the huge (topic x depth x prior-knowledge x format x time-budget) input space so evals generalize rather than overfitting to a few popular topics?
- Can simulated 'learner' agents (AgentA/B-style) produce engagement/preference signal that actually predicts real-learner outcomes, or only catch gross failures? Needs validation against a holdout of real sessions.
- How do we evaluate the interactive visualization itself (does it teach?) beyond 'it renders'? This likely needs a multimodal judge on a screenshot/recording plus interaction checks, which is largely unsolved.
- What's the minimum number of pairwise comparisons (human + LLM) per workflow A/B to reach significance given expected small effect sizes on learning outcomes, and is online traffic sufficient?
- Sequential testing / peeking discipline: how do we run continuous workflow A/B without inflating false positives (always-valid inference vs fixed-horizon)?

## Output-side data collection & learning analytics  
*overall trust: high*

For topic-synthesis, the output side (telemetry from generated interactive learning artifacts) is what makes the founder's other three requirements actually function: workflow A/B testing needs an outcome metric, and the analysis/synthesis evals need ground-truth quality signals from real learners. The good news is that two mature, real interoperability standards already model exactly the events these artifacts produce: xAPI (Experience API, actor-verb-object statements stored in a Learning Record Store) and 1EdTech Caliper Analytics 1.2 (15 metric profiles covering Reading, Media, Assessment, Grading, Feedback, etc.). Both map cleanly onto the signals you care about: dwell/visibility, scroll/section depth, visualization interaction, quiz/comprehension performance, ratings, and revisits. The strongest architectural move is to define a single internal event schema (following durable product-analytics conventions: fixed-string event names like reading:section_view, snake_case typed properties, IDs in properties never in names) that can be projected to BOTH xAPI and Caliper on export, while feeding an internal warehouse. Capture must be instrumented inside each standalone HTML page using Intersection Observer (section/visualization visibility and dwell), the Page Visibility API (true active time, not wall-clock), and navigator.sendBeacon on visibilitychange (reliable delivery on unload). The production-to-eval loop is real and tooling-supported today: Braintrust converts production traces to eval/regression cases with one click and shares one scorer library across logs, experiments, and CI; LangSmith offers online evals plus annotation queues. The key design insight is that learner outcomes (quiz pass rate on a generated comprehension check, section completion, explicit ratings) become the labels that turn a subjective "is this synthesis good?" eval into a measurable one, and the unit of A/B testing should be the whole workflow variant stamped onto every emitted event. Privacy is the main constraint: under GDPR, intervention-grade processing needs explicit, purpose-specific consent, so the schema must be consent-aware and support anonymous/pseudonymous learners by default.

- **Accurate engagement capture for standalone HTML/Canvas/SVG pages requires three specific browser APIs, not naive timers or GA4 defaults: Intersection Observer, the Page Visibility API, and navigator.sendBeacon.** _(confidence: high)_

  GDPR requires explicit consent before processing, with the consent agreement stating purpose, data types, and how/when/where data is processed, plus transparency about storage, retrieval, analysis and use.   → *so what:* Default every learner to an anonymous/pseudonymous id (random per-session or per-device, no PII), store consent_status as a first-class event property, and gate any personalized/identified processing behind explicit opt-in. ### Recommendations
- Define ONE internal event schema as source of truth, borrowing Caliper/xAPI vocabulary: a small fixed set of events (reading:section_view, reading:page_revisit, media:viz_interact, assessment:item_answer, feedback:rating_submit, session:page_dwell, page:completed) with typed snake_case properties. Put topic, page_slug, workflow_variant_id, generation_run_id, learner_pseudo_id, and consent_status in PROPERTIES, never in event names. Validate at ingestion with JSON Schema (Segment-Protocols style) since generated markup varies.
- Ship a single reusable telemetry module that the generation template injects into every standalone page: one IntersectionObserver per section and per visualization (section-view + viz-dwell + scroll-depth), a Page-Visibility-gated active-time timer, and a sendBeacon flush on visibilitychange. Standardizing this guarantees cross-artifact comparability, which is a precondition for valid A/B tests.
- Make the synthesis agent emit an embedded comprehension check (retrieval-practice quiz) per page/section as a first-class artifact, and instrument item-level responses (correct/incorrect, distractor chosen, attempts, time-to-answer). Use quiz pass-rate and per-item error distribution as the primary objective outcome metric for both workflow A/B tests and synthesis-quality evals.
- Adopt a dual-sink architecture: emit each event once, fan out to (1) an LRS / xAPI statement store (portable, standards-clean learning records) and (2) a warehouse or product-analytics store (PostHog/Segment-to-warehouse) optimized for A/B aggregation and eval-dataset construction. Project the internal schema to xAPI and/or Caliper on export; do not couple experimentation velocity to LRS query ergonomics.
- Stamp workflow_variant_id and generation_run_id (sharing trace IDs with the generation pipeline) onto every emitted learner event, with a deterministic variant-assignment key, so a variant's quality is computable as the aggregate of its artifacts' learner outcomes (completion, mean quiz score, mean rating, revisit rate). This is the only reliable way to attribute downstream learner outcomes to whole workflows.
- Wire the production-to-eval loop with Braintrust (or LangSmith): auto-surface low-engagement / high-quiz-failure pages as candidate eval cases, run an SME annotation queue to build a labeled analysis/synthesis-quality dataset, and convert those into regression tests that gate future workflow changes via eval thresholds on PRs.
- Design privacy in from day one: anonymous/pseudonymous learner by default (no PII), consent_status as a first-class event property, aggregate/de-identified engagement under legitimate interest, and explicit purpose-specific opt-in before any identified or intervention-grade processing. Maintain a consent-purpose registry and explicit data-retention windows. Reference the LAK ethics checklists.
- Define the metric hierarchy explicitly before launch: leading engagement signals (section-view rate, viz-interaction rate, active dwell, scroll/section depth) vs. lagging learning signals (quiz pass-rate, completion, explicit rating, revisit/return). Treat the lagging signals as the eval ground truth and the leading signals as fast diagnostics for WHERE in a synthesis learners disengage.

### Open questions
- Primary standard choice: lead with xAPI (lighter, custom-verb-friendly, better fit for bespoke web artifacts) and export Caliper, or vice-versa? Or skip both initially and adopt only on first LMS/partner integration to avoid premature standardization overhead?
- How is learner identity handled across pages and sessions when most learners are likely anonymous? A per-device pseudonymous id enables revisit/cohort analysis but raises consent/retention questions — what is the default, and when does it upgrade to an account?
- What is the minimum traffic/sample size per workflow variant to reach statistical significance on lagging metrics like quiz pass-rate, given that generated topics are long-tail and each artifact may get few learners? Does this force topic-bucketed or surrogate-metric experimentation?
- Where do explicit ratings and free-text feedback sit relative to comprehension checks in the eval ground truth — how are subjective satisfaction and objective comprehension reconciled when they disagree (a beloved page learners fail the quiz on)?
- Should comprehension-check item difficulty/validity itself be evaluated (a quiz that everyone passes or everyone fails is uninformative)? This creates a meta-eval on the synthesis agent's own assessment generation.
- Self-host an open LRS (Learning Locker / Yet SQL LRS) vs. a managed product-analytics platform (PostHog) as the initial single sink — what is the cost/latency/queryability tradeoff at early scale, and does dual-sink justify its complexity on day one?
- What is the data-retention and right-to-erasure design given GDPR, especially when production learner events have been converted into permanent eval regression cases (does erasure propagate into the eval datasets)?

### Verification verdicts

- [mostly-supported] Claim 4: Use sandbox="allow-scripts" WITHOUT allow-same-origin; combining the two negates the sandbox because the framed doc can remove its own sandbox attribute.  
  I tried to refute this and could not — it is the well-documented consensus. For a doc served cross-origin, allow-same-origin instead grants the frame its OWN real origin (cookies/storage/same-origin XHR to its server), which is a different and still-real risk. ## End-to-end technical architecture & orchestration  
*overall trust: high*

For topic-synthesis, the central architectural problem is running a comprehensive multi-agent analysis+synthesis job that can take many minutes (the Claude API skill itself warns single Fable/Opus requests "can run many minutes" and to "plan timeouts/streaming/progress UX"), then streaming progress to a modern UI, persisting generated interactive HTML artifacts plus experiment/eval/telemetry data, and making whole-workflow versions reproducible and A/B-testable. The strongest, most buildable stack for a solo/AI-built repo bootstrapped from agentic-seed is: (1) a durable workflow engine as the spine — Inngest or Trigger.dev (TypeScript-native, serverless-friendly, step memoization so an expensive LLM step never re-runs on retry, no task timeouts) rather than a raw job queue (BullMQ) or self-hosted Temporal, which is more capable but heavier ops for one builder; (2) the Anthropic Claude API directly with `claude-opus-4-8` for the analysis/synthesis agents, orchestrated as code-controlled workflow steps (the "workflow" tier from the claude-api skill) rather than open-ended Managed Agents, because you want deterministic, versionable, replayable control over the pipeline; (3) SSE for one-way progress streaming to a Next.js UI (simpler than WebSockets, Vercel-compatible, auto-reconnect), with the durable engine's own realtime layer as the transport; (4) Postgres (with JSONB for flexible configs/eval-results/telemetry) as the system-of-record plus object storage (Cloudflare R2 / S3) for the generated standalone HTML curriculum pages; (5) a single immutable "workflow version" record — a content-addressed snapshot of the entire pipeline config (DAG, every prompt, model IDs, params, tool set) — that doubles as the A/B assignment unit and the reproducibility key. A/B assignment should live server-side using deterministic hashing (GrowthBook/Statsig pattern) keyed on the workflow-version ID, recorded on every job. Evals (the founder's explicit priority on analysis+synthesis quality) are best run through Braintrust or LangSmith with trajectory-level scoring, since agents judged only on final output "pass 20-40% more test cases than trajectory-level evaluation reveals." The one hard caveat: prompt caching is per-model and model/tool changes invalidate the whole cache, so workflow versioning must treat model+tool+prompt as a single cache-and-reproducibility unit. Confidence is high on the component choices and medium on specific vendor selection (several are viable substitutes).

### Key findings

- **A durable workflow engine — not a raw job queue — is the right spine for long-running multi-agent analysis+synthesis jobs, because durable execution memoizes each step so an expensive LLM call runs exactly once even across crashes and retries.** _(confidence: high)_  
  Job queues (BullMQ) guarantee a worker receives the job but not what happens inside it; long LLM calls can even trip BullMQ's default 30s stalled-job lock. Durable execution engines (Temporal, Inngest, Trigger.dev) persist execution state as the workflow progresses, so on crash they replay from the last successful checkpoint rather than re-running everything — Inngest is explicit that this matters for LLM calls because 're-execution means re-paying for tokens,' and Trigger.dev tasks 'have no timeouts and survive crashes via durable execution, so agent runs can last minutes or hours.' Steps run once, cache their result, and retry independently. This directly maps to a topic-synthesis pipeline of analysis → planning → per-page synthesis fan-out, where each Claude call is costly and you never want to re-pay for a completed step when a later step fails.  
  → *so what:* Make the workflow engine the backbone of the job runtime. Model the pipeline as discrete steps (analyze topic → build curriculum sitemap → fan out one synthesis step per page → assemble hub), each wrapped so its Claude result is memoized. This gives free crash-resumption, per-step retries, and exactly-once expensive calls without hand-rolling a state machine.

- Adopt a durable workflow engine (default Trigger.dev or Inngest) as the job runtime spine; model the pipeline as discrete memoized steps: analyze topic → produce curriculum sitemap (structured-output schema) → parallel fan-out one synthesis step per page → assemble hub. - Use the Claude API workflow tier with claude-opus-4-8 and structured outputs for the analysis sitemap and synthesis metadata; keep all orchestration in your code (not Managed Agents) so the entire DAG is diffable, versionable, and replayable.

- Persist a relational core in Postgres (jobs, workflow_versions, pages with built/soon status, ab_assignments, eval_runs) with JSONB columns for evolving configs/scores/telemetry; store each generated standalone HTML page in R2/S3 keyed by job+page hash.
