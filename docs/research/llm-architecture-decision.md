## topic-synthesis LLM layer: multi-provider architecture decision (rev. 2)

**Date:** 2026-06-17 · **Scope:** `src/llm/` (`client.ts` `complete()`, `models.ts` `STAGE_MODELS`, `pricing.ts`), `src/store/schema.sql`, `src/domain/stages.ts` · **Ground truth read from the repo, not assumed.** · Changes vs rev. 1 are flagged inline as **[rev]**.

### Verified current state (from the code, not the brief)
- `complete()` (`src/llm/client.ts`) returns `{ text, usage, costUsd, stopReason }`, sets `thinking:{type:'adaptive'}` + `output_config:{effort}`, supports a `cache_control` system breakpoint, and throws on `refusal`/`max_tokens`. It has **no structured-output mechanism today** — its own comment (line 61) says "every downstream stage parses structured output and trusts this wrapper," i.e. callers parse `text`. (`src/llm/client.ts:37-83`)
- `STAGE_MODELS` is `Record<Stage, ModelId>` where `ModelId` is a bare Claude string (opus/sonnet/haiku). Stages: `planner, researcher, graph, spec, code, critic`. (`src/llm/models.ts:15-24`)
- `pricing.ts` owns the USD map + `estimateCostUsd()` with Anthropic cache multipliers (read 0.1x, write 1.25x). Its header comment is the staleness liability: *"Rates from the claude-api skill (cached 2026-06-04); update when Anthropic pricing changes."* — an unowned manual cadence. (`src/llm/pricing.ts:1-15`)
- **Eval seams already exist in the DB:** `workflow_version.model_snapshots JSONB`, `run.cost_usd NUMERIC(12,6)`, `run.eleatic_run_id`. **No per-call trace table** — only the `run`-level aggregate. The version-as-arm concept is already encoded (`workflow_version.id` is a content hash of pipeline shape). (`src/store/schema.sql:9-24`)
- **The exact stage output types** (`src/domain/stages.ts`) — load-bearing for the Zod feasibility check below: `PrereqGraph.edges: Array<[from,to]>` and `GatedGraph.{edges: Array<[from,to]>, topoOrder: string[]}` use **tuple** and array shapes; `CritiquedArtifact extends PageArtifact` so it **embeds free-text `html`** alongside typed `passed:boolean`/`critique:string`. Researchers fan out (`Research` is per-question). These are the shapes any structured-output layer must actually carry.
- The pipeline stages are pure `(input)=>Promise<output>` contracts and are **not yet wired to `complete()`**. This bounds the migration: **zero existing call sites to rewrite.** But "zero callers to break" is *not* "small amount of new code" — see Migration below, which is the bulk of the work. **[rev: this conflation was the draft's main overclaim.]**

### TL;DR recommendation
**Adopt the Vercel AI SDK v6 (`ai@6.0.208`) as the provider-neutral LLM layer**, with a `createProviderRegistry` of direct provider SDKs (`@ai-sdk/anthropic@3.0.85`, `@ai-sdk/openai@3.0.73`, `@ai-sdk/google@3.0.83`, `@ai-sdk/openai-compatible@2.0.51` for local/vLLM/Ollama). Keep `pricing.ts` as the synchronous **cost-of-record**, generalized to `Record<"provider:model", pricing>`, and give it an **owner + refresh trigger** (below). Capture per-call `{provider, model, usage, costUsd}` via **one `wrapLanguageModel` middleware** into a new `llm_call` trace table that feeds eleatic. Make structured output uniform with `generateText({ experimental_output: Output.object({ schema }) })` + a Zod re-validate/repair net. Treat **claudeception as a dev-only escape hatch via `claude -p` subprocess**, not an in-pipeline transport — Anthropic's Feb-2026 ToS forbids routing app/Agent-SDK requests through Pro/Max OAuth, so a subscription cannot back any provider in the pipeline.

This wins because **a workflow_version arm becomes mostly data** (`{stage: {provider, model, params}}`), the call site is identical across Claude/GPT/Gemini, and the typed-JSON contract per stage is authored once. The AI SDK normalizes the three genuinely-different native structured-output dialects (Anthropic tool-forced JSON, OpenAI `json_schema` strict, Gemini `responseSchema`/`responseJsonSchema`) behind one `Output.object` call. **[rev]** Honesty correction: an arm does **not** differ *only* in `{provider, model}` — Anthropic-only knobs (effort/thinking/`cache_control`) live in `params` and have no portable equivalent, so an Anthropic-vs-GPT arm necessarily differs in `params` too. The right framing is *"the call site and schema are identical; the arm config is `{provider, model, params}` where `params` is partly provider-specific."* That is still a serializable, hashable axis — just not a two-field one.

**Runner-up: a self-hosted LiteLLM gateway behind a near-unchanged `complete()`.** It wins when the dominant requirement is *authoritative USD cost handed to you* (LiteLLM `x-litellm-response-cost` response header; OpenRouter inline `usage.cost`) with less pricing-map maintenance, **or** when you want minimum churn — LiteLLM's Anthropic-compatible `/v1/messages` surface lets the existing `@anthropic-ai/sdk` wrapper keep working by changing only `baseURL`. The costs: it's infra you run/secure/upgrade (LiteLLM is a Python proxy + DB); cost-read ergonomics differ per gateway (header vs body); gateway USD can carry markup; and it does **not** solve claudeception (gateways are an API-key plane). I rank it second because topic-synthesis is a Next.js/TS app where a unified typed call surface and a serializable per-stage config axis outweigh gateway-reported USD, and a correct pricing map already exists. **[rev]** Note the gateway does *not* eliminate the staleness problem for self-hosted LiteLLM — LiteLLM's own cost map can lag provider price changes too; only the *hosted* OpenRouter path self-reports per-call.

**Not recommended as primary: roll-our-own native abstraction.** Lowest abstraction tax, closest to today's code, a fine *fallback* — but you'd reimplement structured-output normalization, per-provider usage mapping, retries/backoff, and SDK-drift maintenance the AI SDK gives you. **[rev]** Its one underrated advantage is **reversibility**: it has no meta-SDK lock-in (see the lock-in risk below). Choose it only if the provider set stays at 3–4 and the team values zero meta-SDK dependency over breadth.

### Why not the "AI SDK + gateway" hybrid as the default — and the trigger to flip it on
The hybrid (AI SDK *plus* a gateway for authoritative USD) is a sound prod option I keep as a **switch**, not the default, because (a) a correct synchronous pricing map already exists; (b) `@ai-sdk/gateway` USD is an async post-hoc lookup and OpenRouter cost carries markup — both make a *less* clean eval ledger than a deterministic recompute; (c) a gateway adds a dependency/trust/markup surface.

**[rev] Concrete flip-on criteria (any one is the trigger, replacing "later when it hurts"):**
1. **Provider count crosses ~5** *or* you need a model family with no first-party `@ai-sdk/*` package — managing N API keys + N pricing entries by hand is the pain the gateway removes.
2. **`pricing.ts` drift causes a wrong eval verdict** — i.e. a re-run with corrected rates flips an arm ranking. That is the empirical signal that self-maintained pricing is no longer trustworthy and you want a self-reporting source.
3. **You need per-key spend caps / virtual-key budgets** as a platform control (e.g. to stop a runaway eval sweep) — a gateway feature you can't cheaply build.
Until one fires, stay direct-SDK.

### The recommended LLM-layer shape (concrete)

**1. The interface stages call** — evolve `CompleteResult` minimally, add a structured variant:
```ts
interface StageModel { provider: 'anthropic'|'openai'|'google'|'local'; model: string;
  params?: { effort?: Effort; maxTokens?: number; temperature?: number; thinking?: boolean }; }

complete(opts): Promise<{ text; usage: TokenUsage; costUsd; stopReason }>          // text/HTML stages
completeObject<T>(opts & { schema: ZodType<T> }):
  Promise<{ object: T; usage; costUsd; stopReason }>                                // typed stages
```
Both resolve `StageModel` → a `LanguageModelV2` via `createProviderRegistry({ anthropic, openai, google, local })` and call `generateText`. Typed stages pass `experimental_output: Output.object({ schema })`. (`generateObject`/`streamObject` are deprecated in v6 per the official 5→6 migration guide — write the new form now to avoid a second migration.)

**2. How a workflow_version pins per-stage model+params** — generalize `STAGE_MODELS` from `Record<Stage, ModelId>` to `Record<Stage, StageModel>`; store the canonical (sorted-key) JSON in the **existing** `workflow_version.model_snapshots` column; it already feeds `workflow_version.id` (a content hash). An A/B arm is a second `workflow_version` row whose `code` stage flips `provider`. No call-site change.

**3. [rev] Zod feasibility — checked against the *actual* stage types, not gestured at.** Authoring is once-per-stage in `src/domain/stages.ts`:
- `Plan`, `Research`, `PageSpec` → flat objects/arrays; trivially `z.object`. Low risk on all providers.
- `PrereqGraph.edges: Array<[from,to]>` and `GatedGraph.edges` → a **fixed-length tuple**, expressed as `z.tuple([z.string(), z.string()])`. This is the one shape to validate per provider: OpenAI strict `json_schema` represents tuples via `prefixItems`+`minItems`/`maxItems`; **Gemini supports `prefixItems` only via the newer `responseJsonSchema` (Nov 2025), not the older `responseSchema`** — so on the Google provider, tuple edges are the cell to test, and if the AI SDK still emits `responseSchema` you must either upgrade the path or model edges as `z.object({from,to})`. **Recommendation: model edges as `{from,to}` objects in the Zod schema** (cheap, lossless, sidesteps all tuple/prefixItems variance) and adapt to the TS tuple at the boundary. `topoOrder: string[]` is a plain array — safe everywhere.
- `GatedNode.route` is a `PageStatus` enum → `z.enum`, safe everywhere.
This is a real, small porting task with one genuine edge (tuples on Gemini), not a hand-wave.

**4. Where usage+cost is captured for eleatic** — one `wrapLanguageModel({ middleware })` (`LanguageModelV2Middleware`) wraps every registry model. In `wrapGenerate`, after the call it reads `result.usage`, normalizes to `TokenUsage` (Anthropic input/output/cache_read/cache_creation; OpenAI prompt/completion + `prompt_tokens_details.cached_tokens`; Gemini promptTokenCount/candidatesTokenCount), computes `costUsd = estimateCostUsd("provider:model", usage)`, and **inserts a row into a new `llm_call` table** keyed by `run_id`+`stage`+`workflow_ver`+`{provider, model, usage_json, cost_usd, raw_usage_json}`. Read usage from the result object, not OTel spans (vercel/ai #12801 documents span gaps).

**[rev] The middleware is a single point of failure and the sole writer of the cost-of-record — treat it as such:**
- **If it throws** after a successful provider call, you've spent tokens with no ledger row. Wrap the insert in try/catch that logs + emits a `cost_trace_missing` marker on the `llm_call` row (insert a row with `cost_usd=NULL, error='middleware'`) rather than dropping it silently — a missing row must be *visible*, because the eval pillar's whole job is cost truth.
- **If it mis-normalizes one provider's usage**, every arm on that provider is silently wrong. Mitigation: store `raw_usage_json` verbatim so any normalization bug is *recomputable after the fact*, and add a unit test per provider asserting the normalizer against a recorded real `result.usage` fixture.
- **If it's bypassed** (a direct `generateText` that skips the registry/wrapper), there's no row at all. Mitigation: make the wrapped registry the *only* exported entry point; lint/forbid importing provider packages outside `src/llm/`.

**[rev] Concurrency / idempotency of the `run.cost_usd` roll-up.** Researchers fan out, so many concurrent `llm_call` inserts feed one `run` aggregate. Do **not** read-modify-write `run.cost_usd` from the middleware (lost-update race). Two safe options: (a) make `llm_call` the source of truth and compute `run.cost_usd` as `SELECT SUM(cost_usd) FROM llm_call WHERE run_id=$1` at read time / run-finalize, never incrementally; or (b) if you must denormalize, update via `UPDATE run SET cost_usd = cost_usd + $delta` (atomic in Postgres) inside the same transaction as the `llm_call` insert. Option (a) is simpler and is the recommendation — the SUM is then trivially correct under fan-out. Give `llm_call` an idempotency key (`run_id`+`stage`+`call_ordinal`) so a retried insert can't double-count.

**5. How structured outputs are made uniform — and what happens at the free-text↔typed boundary.** Author each stage's Zod schema once; pass to `Output.object`; **always re-validate the returned object against the same Zod schema in the wrapper** and do one bounded repair-retry on failure. This preserves the load-bearing invariant ("downstream stages trust the wrapper") across providers.

**[rev] The `code`→`critic` boundary is the sharp edge and gets an explicit contract.** `code` emits free-text HTML; `critic` consumes a `PageArtifact` (which embeds `html`) and emits a typed `CritiquedArtifact` (`extends PageArtifact`, adding `passed`/`critique`). So a malformed/oversized HTML artifact flows *into* a typed stage. Contract: (a) `code` stays a **text** stage (`complete`, not `completeObject`) — forcing `Output.object` on HTML adds a failure mode for no benefit; (b) before `critic`, validate the `PageArtifact` shape (non-empty `html`, valid `nodeSlug`, `spec` present) and fail loud on a malformed artifact rather than feeding garbage into the typed critic; (c) `critic`'s typed output carries the *same embedded `html`* forward — re-validating `CritiquedArtifact` therefore re-checks the HTML field is still present/unmangled, which catches a provider that silently drops or escapes it.

**[rev] Caveat to verify before shipping Anthropic typed stages** — the root cause is real and now has two distinct failure modes:
- **tool-forcing vs thinking:** Anthropic structured output forces `tool_choice`, but extended thinking requires `tool_choice: auto|none` — verified open conflict (vercel/ai #7220, with #9351 documenting a middleware workaround that sets `tool_choice:auto` while still passing the JSON tool). Your wrapper uses **adaptive thinking**, so this *will* bite typed Anthropic stages. Test `Output.object` on the Anthropic provider with thinking on **first**, and budget for the #9351 middleware workaround.
- **keyword stripping (vercel/ai #13355):** `@ai-sdk/anthropic` forwards Zod-emitted JSON-Schema validation keywords the Anthropic API rejects with a 400 (the native SDK strips them). Keep your Zod schemas to the supported keyword subset, or post-process the schema. This is a *separate* bug from the thinking conflict and hits even with thinking off.

### Migration from current → recommended (the work, sequenced) **[rev — was missing entirely]**
This is a single PR's worth of *new* code (zero callers to break), but it is not small. Ordered so each step is testable in isolation:
1. **Add packages + pin** (`ai@6.0.208`, the four `@ai-sdk/*@3.x`/`@2.x` above). ~½ day incl. the "ai v6 + providers v3" version-skew sanity test.
2. **Generalize `pricing.ts`** key from `model` to `"provider:model"`; add OpenAI/Gemini/local entries; keep cache multipliers Anthropic-only. ~½ day. (Owner/refresh: next section.)
3. **Build the registry + middleware** (`createProviderRegistry`, `wrapLanguageModel`, usage normalizers + per-provider fixture tests, the throw/bypass/raw-usage safeguards from §4). **This is the largest piece — ~2–3 days** including the normalizer fixtures.
4. **Add the `llm_call` table** + migration; switch `run.cost_usd` to a `SUM(llm_call)` read (or atomic-delta). ~1 day incl. the concurrency test. *(Open question #3: confirm whether eleatic reads `llm_call` directly or only via `run.eleatic_run_id` — see Decision Questions; this gates the table's foreign-key/ingestion shape.)*
5. **Port the six stage types to Zod** in `src/domain/stages.ts`, modeling edges as `{from,to}` objects (§3). ~1 day.
6. **Implement `complete`/`completeObject`** over the registry + Zod re-validate/repair net; keep the `code` stage on `complete`. ~1 day.
7. **Anthropic typed-stage spike** (the #7220/#9351 thinking workaround and #13355 keyword stripping) — do this **before** declaring the design done, because if Anthropic typed-output-with-thinking can't be made reliable, `graph`/`spec`/`critic` on Anthropic need a fallback (thinking-off, or text+parse). ~1 day, high-information.

**Rough total: ~7–9 engineer-days**, dominated by the middleware (3) and the Anthropic spike (7). "Small blast radius" is true only in the call-graph sense; the build is real.

### Cost-of-record ownership + refresh **[rev — was an open gap]**
`pricing.ts` is the eval cost-of-record, so stale rates *silently flip arm rankings*. Assign:
- **Owner:** the person who owns the eval pillar (single named owner in `pricing.ts` header, replacing the passive "update when Anthropic pricing changes").
- **Source of truth per provider:** Anthropic/OpenAI/Google public pricing pages (cite the URL + check date in-file, as the file already does for the 2026-06-04 Anthropic snapshot).
- **Cadence + alarm:** (a) a `pricing.test.ts` assertion that the in-file `cached_at` date is < 90 days old, failing CI when stale — this is the *alarm* the code comment lacks; (b) re-snapshot on any provider price-change announcement. (c) Because every `llm_call` stores `raw_usage_json`, a corrected rate lets you **recompute historical eval costs** — so a missed update is recoverable, not destructive, as long as raw usage is retained.

### Risks worth pricing in
- **[rev] Middleware is the cost-ledger SPOF** — covered with throw/bypass/mis-normalize mitigations in §4; the residual risk is an *undetected* normalizer bug on a low-traffic provider, bounded by retaining `raw_usage_json` for recompute.
- **[rev] Structured-output retries contaminate the eval comparison.** If weak/local-model arms fail `Output.object` and hit the repair-retry path more often, they burn extra tokens + latency that land in the very cost/quality ledger the eval pillar compares — so "apples-to-apples" is *not* free. Defend it: record `repair_attempts` and `repaired:boolean` on each `llm_call` row, count repair tokens in `cost_usd`, and report repair rate as a first-class arm metric (a high repair rate is itself a quality signal, not noise to hide).
- **Provider-package version skew:** `ai` is v6 but providers are v3 (`@ai-sdk/anthropic@3.0.85`, `@ai-sdk/openai@3.0.73`, `@ai-sdk/google@3.0.83`, `@ai-sdk/openai-compatible@2.0.51`, `@ai-sdk/gateway@3.0.133`; verified `npm view` 2026-06-17). Pin and test together — correct but surprising.
- **[rev] Meta-SDK lock-in / reversibility.** Adopting the AI SDK couples the LLM layer's upgrade cadence to a fast-moving meta-SDK (generateObject deprecated within one major). Exit cost is bounded — the surface area we depend on is small (`generateText` + `Output.object` + `wrapLanguageModel` + registry) and the roll-our-own fallback is a known quantity — but it's a real coupling to weigh, not zero.
- **Anthropic-specific knobs don't generalize:** adaptive thinking, `output_config.effort`, `cache_control` move into `providerOptions.anthropic` with no portable OpenAI/Gemini equivalent. They live in `StageModel.params`, which is why an Anthropic arm and a GPT arm differ in more than `{provider,model}`.
- **[rev] claudeception cost is an estimate AND a split-brain hazard.** `claude -p --output-format json` returns `total_cost_usd`, but Anthropic's docs say it's a *client-side estimate from a bundled price table*, not authoritative billing. The draft implied this could be "reconciled" into the eleatic trace — but mixing estimate-sourced USD (dev) and `pricing.ts`-sourced USD (CI/prod) in one ledger is the same split-brain hazard called out for gateways. Fix: tag every `llm_call` row with `cost_source ∈ {pricing_ts, claude_p_estimate, gateway}` and **never compare arms across cost sources**; dev-estimate rows are for local sanity only, excluded from eval verdicts.
- **[rev] Subscription-path rate-limit lock-in (DEV).** `claude -p` is ToS-safe for local dev, but it draws from the developer's *normal* Pro/Max subscription limits (the separate-credit-pool change was *paused* 2026-06-15, never enacted). A dev running many pipeline iterations can exhaust their personal limits and **block their own interactive Claude Code** — an operational self-DOS, not just a billing footnote. Mitigation: cap dev-loop iterations, or point heavy local sweeps at a cheap real API key.
- **Local/open models' strict JSON is best-effort** — the Zod re-validate/repair net is mandatory there, and (per the retry risk above) their repair rate is a tracked metric.
- **Split-brain USD if you later add a gateway** — gateway-reported vs `pricing.ts` can disagree (markup, cache discounts). One `cost_source` per arm; always store `raw_usage_json` so any number is recomputable.

### Open decision questions (highest leverage)
1. **Cost-of-record:** self-computed `pricing.ts` (deterministic, you maintain rates + the 90-day staleness alarm) vs gateway-reported USD (self-reporting but async/markup)? Recommendation: self-computed now; gateway only when a flip-on criterion fires.
2. **Direct provider SDKs vs gateway in prod:** do you need one-key access / virtual-key budgets / 100+ models enough to take on a gateway dependency + markup? If not, direct SDKs + registry is simpler.
3. **[rev — now load-bearing for the migration] How does eleatic ingest `llm_call`?** Direct read of the table, or only via `run.eleatic_run_id` → run-level rollup? This gates step 4 of the migration: a direct read needs the table's columns to match eleatic's per-call schema; an indirect path lets `llm_call` stay app-private and only the SUM crosses to eleatic. **Resolve before building the table.**
4. **Which stages need typed output vs free text?** `graph`/`spec`/`critic` need typed JSON; `code` emits HTML (free text, stays on `complete`). Confirmed by the stage types; the `code`→`critic` boundary contract is in §5.
5. **claudeception transport for dev:** `claude -p` subprocess (ToS-safe, separate process, estimate-cost, draws on personal limits) vs `ANTHROPIC_BASE_URL` proxy (zero code change but ToS-sensitive if it forwards OAuth)? Recommendation: subprocess; get it in writing if any shared/CI path could touch it.
6. **How are non-portable Anthropic knobs (effort/thinking/caching) represented in a workflow_version** so an Anthropic arm and a GPT arm stay comparable — drop them on non-Anthropic arms, or map "effort: high" to a provider-equivalent notion? (This is the residue of the "differs in more than {provider,model}" correction.)
