## Color & contrast
Dark, technical palette (inspired by the ai-concept-viz explainers). Body `--text` on `--bg-app` clears WCAG AA for normal text; muted text used only at ≥ `--fs-body`. Status is always conveyed by **label + icon**, never color alone. Target: WCAG 2.2 AA.

## Typography
Three families, all declared in §0. Fonts are **chrome**: Inter (`--sans`) and JetBrains Mono (`--mono`) are LOADED once at the app boundary via `next/font/google` in `src/app/layout.tsx` (which puts `--font-inter` / `--font-jetbrains-mono` on `<html>`, picked up by the §0 stacks) — never imported in core code; `next/font` self-hosts the files and emits a metrics-matched fallback with `display: swap`, so the chrome shows real Inter without a layout-shift (FOUT, not CLS). **Chrome + body** use the sans stack (`--sans`, leading with loaded Inter): the intake form, top bar, library, generating-state — everything that isn't a lesson heading. **Lesson headings** (the lesson `h1`/`h2`) use the serif stack (`--serif`, ADOPTED in TS-3 from the v11 mock; its explicit Georgia→Times fallback keeps a serif voice on Linux/Cloud Run where the Apple reading serifs are absent) — a reading voice distinct from the chrome's sans. **Code/tokens, eyebrows, fine-print, and the generating timeline** use the monospace stack (`--mono`, ONE token spanning the SF-Mono / JetBrains-Mono split — Figma uses SF Mono for eyebrows/fine-print and JetBrains Mono for the generating timeline; the loaded JetBrains Mono guarantees the mono voice off-Apple); the per-step timer/durations use it with tabular figures so digits don't jitter (§Components → Progress). Scale per §0. **Line-height is per-role** (§0 `--lh-*`, superseding the old single "1.5 body / 1.2 headings" pair): `--lh-reading 1.7` for reading prose, `--lh-display 1.06` / `--lh-heading 1.18` for the serif lesson headings, `--lh-gloss 1.55` for the gloss/caption band. **Letter-spacing is per-role** too (§0 `--ls-*`): tight-negative on display/headings, positive on the uppercase-mono eyebrows + fine-print.

## Motion
Transitions use the **`transitions-dev`** snippet catalog (copy-paste CSS, no runtime dependency) at the §0 durations and the single §0 `--ease` (the `--dur-fast`/`--dur-base`/`--dur-slow` tiers + `--ease`; this section states no literal duration of its own — §0 wins, and the motion catalog consumes the resolved 120/200/440 + `--ease`). **The catalog is LANDED (TS-24):** `src/app/globals.css` `:root` declares a small set of NAMED transition primitives (`--tr-color`/`--tr-bg`/`--tr-hover` @ `--dur-fast`, `--tr-progress`/`--tr-state` @ `--dur-base`) that compose ONLY `var(--dur-*)` + `var(--ease)` — so EVERY animated surface references a primitive by name and the value lives in exactly one place (no per-component duration/easing literal). Surfaces on the catalog: the library card hover (`--tr-hover`), the primary button + sign-out (`--tr-bg`/`--tr-color`), the reading-progress fill (`--tr-progress`), the generating rail's state change (`--tr-state`); the loop spinner reuses the `--dur-slow` tier with `linear` (a continuous rotation can't ease its loop seam); the card→reader morph reads `--dur-slow`/`--ease` in its View-Transition block. Allowed in the skeleton: staggered hub-tile reveal, the **generating timeline's staggered step reveal** (each rail row enters via the `rail-reveal` keyframe @ `--dur-base`, delay = row-index × `--dur-fast`), modal/panel open, status-badge change, progress updates. The **card→reader box-FLIP morph** is the one exception that rides a different transport: the cross-document `@view-transition { navigation: auto }` mechanism, which activates ONLY on a real full-document navigation — so the FLIP-origin card MUST link via a plain `<a>` (NOT `next/link`, whose client-side soft nav would never fire it; §Components → Library). **Reduced motion:** `@media (prefers-reduced-motion: reduce)` removes non-essential transitions and staggers (it zeroes `animation-duration` AND `animation-delay`, so the timeline reveal shows instantly with no stagger); status/progress change instantly.

## Lesson layout (LOCKED — the design target / acceptance bar)

The **LOCKED** design target the generated single-lesson page is meant to be built against — the acceptance bar a future workspace-rendering pass builds toward (the v11 graded critic that once gated the artifact on this bar was REVERTED to the binary blob critic — see `docs/plans/lesson-workspace.md` — so today no pipeline stage emits or gates against this layout; the live binary `critic` judges generic teaches-goal/interactive/a11y/self-contained, NOT this workspace layout). This section wins on any lesson-layout conflict and is the single source of truth (it supersedes the scratch ledger, now promoted here — see below). The reference implementation is the scratch mockup `.superpowers/topic-synthesis-lesson-workspace-v11.html`. The bar legitimately survives the revert as the design target a future pass renders to; *regression vs. the best prior version* would be an **offline** eleatic `--baseline` bench comparison, never an in-run critic gate.

**The workspace.** A lesson is a two-column **workspace**, never a page-container/void: a frozen reading column (measure ~62ch) plus an always-full **apparatus panel** docked beside it. The whole reading+panel assembly is capped (~1500–1640px) and centered with equal gutters; "use the area" is satisfied by the full apparatus column, not by stretching prose to the viewport edge. Surfaces use the dark §0 tokens; the card↔reader morph lives on a single `#readerPanel.morph-box` wrapper that wraps the lesson iframe and is the morph's FLIP **destination** (it carries a `view-transition-name` pairing with the library card's FLIP origin). The morph is **box-only** — the container box's geometry animates while the opaque iframe contents render at the final frame (the wrapper is built in TS-20; the animation lands in TS-21).

**Four locked decisions:**
1. **Densify every section.** Each section carries a curated apparatus stack — key-term glosses (term dotted in prose, definition in the panel), a where-am-I/progress cue, a teaching mini-figure where it helps, live readouts when a component runs, sources where they exist. The working column is full on *every* section — full **because each section carries apparatus, NOT because citations exist** (typically only a fraction of sections cite a source). Apparatus must *add* what the prose doesn't already state — never filler — capped at ≤3 glosses + ≤1 mini-figure per section to avoid a dashboard.
2. **Lone element centers its text.** A genuinely standalone element with no apparatus beside it sets its text `text-align:center`, applied at the block/section level — never per-paragraph alternation inside a paired section.
3. **Cap + center the assembly.** Capped width, equal gutters, **nothing pinned to the true viewport edge** — scrubbers and controls live inside the capped frame, not against the screen edge.
4. **Stable spine (HARD rule — wins all conflicts).** The reading column holds the *exact* same horizontal position and width for every paragraph and across section boundaries. A lone element center-aligns its text *within* the fixed column; it never moves the column. Zero left-right jitter.

**Named grid-line set (design direction — not yet pipeline-enforced).** The lesson workspace is a CSS grid whose canonical column track names are `[screen-start] [read] [gap] [panel] [scrub]`. This is the abbreviated reference set; the literal lives **here** (which wins on design conflict). The `[scrub]` track is reserved for the in-iframe dot-scrubber rail, parked inside the capped frame so the scrubber is not an orphaned track and nothing is pinned to the true viewport edge. (A prior v11 pipeline rework had the synthesis `code` stage emit this grid and a graded critic gate on `[scrub]`; that rework was REVERTED to the blob-binary pipeline — see `docs/plans/lesson-workspace.md` — so today no pipeline stage emits or gates this literal. It remains the design target for when a future workspace-rendering pass is implemented.) The v11 reference mock materializes a fuller line set (`[screen-start] [read-start] … [read-end] [panel-start] … [panel-end] [scrub] … [screen-end]`, with an *unnamed* `var(--col-gap)` track between `[read-end]` and `[panel-start]` — there is no `[gap]` named line in the mock) as the reference implementation; the abbreviated names above are the canonical design-target set (the critic that once gated them is reverted, per above), not the only legal track labels.

**Per-section composition** (decided per section, not one global pick):
- **With apparatus** (≈all sections post-densify): a 2-column unit — frozen prose spine on the left (text left-aligned), apparatus docked in the right panel track beside it.
- **Without apparatus** (rare): prose stays on the same fixed spine, text center-aligned, panel track absent — the spine does not move.

**Invariants:** two columns max — never one, never three. Interactive components (the Play interactive) get a full-width bounded plate, fully visible, with no prose overlap. Citations render beside their text in the panel, never a lone card in a void. Responsive: collapse to one column ≤~900px (apparatus reflows directly under the prose it annotates); no horizontal overflow at 390 (the topbar fits within the viewport; the wordmark is hidden ≤640). Reduced-motion + a11y are correctness, not polish (§Motion, §Accessibility).

**Anti-patterns (rejected — must not recur):** reserved/empty margin · single column · prose-over-component occlusion · per-paragraph horizontal jitter · lopsided/left-pinned prose with dead right · an edge-pinned lone element at wide viewports · clipped figures or labels.

## Accessibility
Target WCAG 2.2 AA. Visible `:focus-visible` ring (2px `--interactive`). Full keyboard operability of the form + hub. Status by label+icon, not color alone. Reduced motion honored (§Motion). Generated artifacts carry their **own** a11y contract (a generation target — see `docs/plans/`); the chrome never depends on an iframe's internals for its own accessibility.
