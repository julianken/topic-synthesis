## Color & contrast
Dark, technical palette (inspired by the ai-concept-viz explainers). Body `--text` on `--bg-app` clears WCAG AA for normal text; muted text used only at в‰Ө `--fs-body`. Status is always conveyed by **label + icon**, never color alone. Target: WCAG 2.2 AA.

## Typography
Three families, all declared in В§0. **Chrome + body** use the sans stack (`--sans`): the intake form, top bar, hub, generating-state вЂ” everything that isn't a lesson heading. **Lesson headings** (the lesson `h1`/`h2`) use the serif stack (`--serif`, ADOPTED in TS-3 from the v11 mock) вЂ” a reading voice distinct from the chrome's sans. **Code/tokens** (and generated pages' code) use monospace (`--mono`); the per-step timer/durations use the mono stack with tabular figures so digits don't jitter (В§Components в†’ Progress). Scale per В§0; line-height 1.5 body / 1.2 headings.

## Motion
Transitions use the **`transitions-dev`** snippet catalog (copy-paste CSS, no runtime dependency) at the В§0 durations and the single В§0 `--ease` (the `--dur-fast`/`--dur-base`/`--dur-slow` tiers + `--ease`; this section states no literal duration of its own вЂ” В§0 wins, and the motion catalog consumes the resolved 120/200/440 + `--ease`). **The catalog is LANDED (TS-24):** `src/app/globals.css` `:root` declares a small set of NAMED transition primitives (`--tr-color`/`--tr-bg`/`--tr-hover` @ `--dur-fast`, `--tr-progress`/`--tr-state` @ `--dur-base`) that compose ONLY `var(--dur-*)` + `var(--ease)` вЂ” so EVERY animated surface references a primitive by name and the value lives in exactly one place (no per-component duration/easing literal). Surfaces on the catalog: the library card hover (`--tr-hover`), the primary button + sign-out (`--tr-bg`/`--tr-color`), the reading-progress fill (`--tr-progress`), the generating rail's state change (`--tr-state`); the loop spinner reuses the `--dur-slow` tier with `linear` (a continuous rotation can't ease its loop seam); the cardв†’reader morph reads `--dur-slow`/`--ease` in its View-Transition block. Allowed in the skeleton: staggered hub-tile reveal, the **generating timeline's staggered step reveal** (each rail row enters via the `rail-reveal` keyframe @ `--dur-base`, delay = row-index Г— `--dur-fast`), modal/panel open, status-badge change, progress updates. The **cardв†’reader box-FLIP morph** is the one exception that rides a different transport: the cross-document `@view-transition { navigation: auto }` mechanism, which activates ONLY on a real full-document navigation вЂ” so the FLIP-origin card MUST link via a plain `<a>` (NOT `next/link`, whose client-side soft nav would never fire it; В§Components в†’ Library). **Reduced motion:** `@media (prefers-reduced-motion: reduce)` removes non-essential transitions and staggers (it zeroes `animation-duration` AND `animation-delay`, so the timeline reveal shows instantly with no stagger); status/progress change instantly.

## Lesson layout (LOCKED вЂ” the standing acceptance bar)

The **LOCKED** spec the generated single-lesson page is built and critiqued against вЂ” the standing acceptance bar the graded critic gates the artifact on and every build diffs against. This section wins on any lesson-layout conflict and is the single source of truth (it supersedes the scratch ledger, now promoted here вЂ” see below). The reference implementation is the scratch mockup `.superpowers/topic-synthesis-lesson-workspace-v11.html`. The in-run critic (TS-7) scores the **current** artifact against this bar only; *regression vs. the best prior version* is an **offline** eleatic `--baseline` bench comparison, **not** an in-run critic gate (program decision 3 вЂ” see the program doc, `docs/plans/lesson-workspace.md`; this section does not restate the program decisions, it points at them).

**The workspace.** A lesson is a two-column **workspace**, never a page-container/void: a frozen reading column (measure ~62ch) plus an always-full **apparatus panel** docked beside it. The whole reading+panel assembly is capped (~1500вЂ“1640px) and centered with equal gutters; "use the area" is satisfied by the full apparatus column, not by stretching prose to the viewport edge. Surfaces use the dark В§0 tokens; the cardв†”reader morph lives on a single `#readerPanel.morph-box` wrapper that wraps the lesson iframe and is the morph's FLIP **destination** (it carries a `view-transition-name` pairing with the library card's FLIP origin). The morph is **box-only** вЂ” the container box's geometry animates while the opaque iframe contents render at the final frame (the wrapper is built in TS-20; the animation lands in TS-21).

**Four locked decisions:**
1. **Densify every section.** Each section carries a curated apparatus stack вЂ” key-term glosses (term dotted in prose, definition in the panel), a where-am-I/progress cue, a teaching mini-figure where it helps, live readouts when a component runs, sources where they exist. The working column is full on *every* section вЂ” full **because each section carries apparatus, NOT because citations exist** (typically only a fraction of sections cite a source). Apparatus must *add* what the prose doesn't already state вЂ” never filler вЂ” capped at в‰¤3 glosses + в‰¤1 mini-figure per section to avoid a dashboard.
2. **Lone element centers its text.** A genuinely standalone element with no apparatus beside it sets its text `text-align:center`, applied at the block/section level вЂ” never per-paragraph alternation inside a paired section.
3. **Cap + center the assembly.** Capped width, equal gutters, **nothing pinned to the true viewport edge** вЂ” scrubbers and controls live inside the capped frame, not against the screen edge.
4. **Stable spine (HARD rule вЂ” wins all conflicts).** The reading column holds the *exact* same horizontal position and width for every paragraph and across section boundaries. A lone element center-aligns its text *within* the fixed column; it never moves the column. Zero left-right jitter.

**Named grid-line set (design direction вЂ” not yet pipeline-enforced).** The lesson workspace is a CSS grid whose canonical column track names are `[screen-start] [read] [gap] [panel] [scrub]`. This is the abbreviated reference set; the literal lives **here** (which wins on design conflict). The `[scrub]` track is reserved for the in-iframe dot-scrubber rail, parked inside the capped frame so the scrubber is not an orphaned track and nothing is pinned to the true viewport edge. (A prior v11 pipeline rework had the synthesis `code` stage emit this grid and a graded critic gate on `[scrub]`; that rework was REVERTED to the blob-binary pipeline вЂ” see `docs/plans/lesson-workspace.md` вЂ” so today no pipeline stage emits or gates this literal. It remains the design target for when a future workspace-rendering pass is implemented.) The v11 reference mock materializes a fuller line set (`[screen-start] [read-start] вЂ¦ [read-end] [panel-start] вЂ¦ [panel-end] [scrub] вЂ¦ [screen-end]`, with an *unnamed* `var(--col-gap)` track between `[read-end]` and `[panel-start]` вЂ” there is no `[gap]` named line in the mock) as the reference implementation; the abbreviated names above are the canonical critic-gated set, not the only legal track labels.

**Per-section composition** (decided per section, not one global pick):
- **With apparatus** (в‰€all sections post-densify): a 2-column unit вЂ” frozen prose spine on the left (text left-aligned), apparatus docked in the right panel track beside it.
- **Without apparatus** (rare): prose stays on the same fixed spine, text center-aligned, panel track absent вЂ” the spine does not move.

**Invariants:** two columns max вЂ” never one, never three. Interactive components (the Play interactive) get a full-width bounded plate, fully visible, with no prose overlap. Citations render beside their text in the panel, never a lone card in a void. Responsive: collapse to one column в‰¤~900px (apparatus reflows directly under the prose it annotates); no horizontal overflow at 390 (the topbar fits within the viewport; the wordmark is hidden в‰¤640). Reduced-motion + a11y are correctness, not polish (В§Motion, В§Accessibility).

**Anti-patterns (rejected вЂ” must not recur):** reserved/empty margin В· single column В· prose-over-component occlusion В· per-paragraph horizontal jitter В· lopsided/left-pinned prose with dead right В· an edge-pinned lone element at wide viewports В· clipped figures or labels.
